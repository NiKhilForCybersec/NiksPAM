Real-World Attack Scenarios

Understanding how attackers actually operate helps you appreciate why PAM and KMS controls exist. This page walks through real attack patterns and shows exactly how proper controls would stop or detect them.

The Attack Lifecycle

Sophisticated attacks don't happen instantly. Attackers follow a methodical process, and understanding this process reveals where PAM and KMS create barriers.

1

Initial Access

Attacker gets a foothold: phishing email, exploited vulnerability, compromised credentials, or malicious insider. At this point, they typically have limited, non-privileged access.

2

Discovery

Attacker explores: What network am I on? What systems exist? Where are the valuable targets? Who has admin access?

3

Privilege Escalation

Attacker tries to gain higher privileges: steal admin credentials, exploit vulnerabilities, abuse misconfigurations. This is where PAM creates the biggest barrier.

4

Lateral Movement

With elevated credentials, attacker moves to other systems: domain controllers, database servers, backup systems. PAM limits how far they can go.

5

Objective Execution

Attacker achieves their goal: steal data, deploy ransomware, establish persistence. KMS encryption can make stolen data useless.

The Key Insight

Almost every significant breach involves privileged credentials. Attackers can do limited damage with a regular user account. Real damage requires admin access. This is why privileged access is the most valuable target—and why protecting it matters so much.

Credential Theft Attacks

Stealing credentials is often the easiest path to escalation. Let's examine common techniques and how PAM defeats them.

Attack: Keylogger Captures Admin Password

The Attack

Attacker installs a keylogger on an admin's workstation (via phishing, drive-by download, or physical access). The keylogger records every keystroke, including when the admin types their password to connect to servers.

Without PAM: Admin types password directly. Keylogger captures "P@ssw0rd123!". Attacker now has the admin's credentials and can use them from anywhere.

With PAM: Admin connects through PAM's session proxy. The admin never types the server password—PAM injects it. The keylogger captures nothing useful. Even if the attacker gets the admin's PAM portal password, MFA blocks unauthorized access, and session proxy means passwords still aren't exposed.

Attack: Memory Scraping (Mimikatz)

The Attack

Tools like Mimikatz can extract passwords and password hashes from Windows memory. If an admin has logged into a compromised server, their credentials may be recoverable from memory.

Without PAM: Admin RDPs to server, types credentials. Those credentials are cached in memory. Attacker runs Mimikatz, extracts the admin's password hash or even plaintext password. Attacker can now impersonate the admin.

With PAM:

  • Admin never types credentials on the target—PAM session proxy handles authentication
  • The admin's personal credentials never touch the target server
  • The privileged account credentials are only on the PAM server, which is hardened
  • Even if credentials are captured, they rotate after use—captured password is already invalid

Attack: Password in Config Files

The Attack

Developers hard-code database passwords in application config files. Attacker gains access to the web server (low-privilege), reads config files, and finds database credentials in plaintext.

Without PAM: Config file contains db_password=SecretPass123. Attacker reads it, connects to database, exfiltrates all data.

With PAM (for application credentials):

  • Application retrieves credentials from PAM via API at startup
  • No credentials in config files, source code, or environment variables
  • Credentials can be rotated without redeploying the application
  • All access is logged—unusual access patterns trigger alerts

Lateral Movement

Once attackers have some credentials, they try to use them on other systems. This is where the same password everywhere becomes catastrophic.

Attack: Pass-the-Hash

The Attack

In Windows environments, attackers don't even need the password—the password hash can be used directly to authenticate (NTLM authentication). Attacker captures hash from one machine and "passes" it to authenticate to other machines.

Scenario without PAM:

  1. Attacker compromises workstation
  2. Local admin account has password "AdminPass123" — same on all workstations
  3. Attacker extracts hash of local admin
  4. Attacker uses hash to authenticate to ALL other workstations (because same password = same hash)
  5. Attacker now controls every workstation in the company

With PAM (unique passwords per system):

  1. Attacker compromises workstation
  2. Local admin password is unique: "x9Kp2#mNv..." (set by PAM)
  3. Attacker extracts hash
  4. Hash ONLY works on this one workstation—every other machine has a different password
  5. Lateral movement is blocked

Attack: Stolen SSH Key Used Everywhere

The Attack

Admin uses the same SSH key pair for all 500 Linux servers. Attacker compromises the admin's laptop, finds the private key file, and now has root access to all 500 servers.

With PAM SSH Key Management:

  • SSH private keys stored in PAM vault, not on admin's laptop
  • Admin connects through PAM session proxy—never touches the private key
  • Or: PAM issues short-lived SSH certificates (expire in hours)
  • Even if a key is somehow compromised, it's rotated regularly

Privilege Escalation

Attackers with limited access seek ways to gain higher privileges. Let's see how PAM makes this harder.

Attack: Exploiting Weak Service Account

The Attack

Attacker discovers a service account that (a) has a weak password, (b) is a member of Domain Admins (because "it needed access to everything"), and (c) the password hasn't been changed in 3 years.

Without PAM: Attacker cracks or guesses the service account password. Instant Domain Admin. Game over.

With PAM:

  • Service account password is 30+ random characters (can't be cracked or guessed)
  • Password rotates every 30 days (or more frequently)
  • Service account is NOT in Domain Admins (PAM enables reviewing and right-sizing permissions)
  • Any use of the service account is logged—unusual login times/sources trigger alerts

Attack: Kerberoasting

The Attack

In Active Directory, any authenticated user can request Kerberos service tickets for any service account with a Service Principal Name (SPN). The ticket is encrypted with the service account's password hash. Attackers request tickets, then crack them offline to recover passwords.

Without PAM: Service accounts have human-chosen passwords like "Summer2023!". Attacker cracks them in minutes.

With PAM:

  • Service account passwords are 25+ character random strings
  • Cracking would take longer than the age of the universe
  • Passwords rotate before any cracking could succeed anyway
  • Better yet: use Group Managed Service Accounts (gMSA) where AD manages the password automatically

Insider Threats

Not all threats come from outside. Malicious or negligent insiders with privileged access can cause massive damage.

Scenario: Disgruntled DBA

The Scenario

A database administrator learns they're being laid off. Before their last day, they want to steal customer data to sell to competitors, and maybe delete some tables to cause chaos.

Without PAM:

  • DBA knows all database passwords (they set them up)
  • DBA can connect directly from home using VPN
  • No one knows what queries the DBA runs
  • After termination, DBA still knows the passwords (which haven't been changed)

With PAM:

  • DBA never knew the passwords—PAM injected them
  • Every SQL query is recorded—mass data export would be visible
  • Real-time monitoring—security team sees unusual activity
  • Termination triggers rotation—all credentials DBA ever touched are changed
  • Forensic evidence—complete audit trail of everything they did

Scenario: Accidental Exposure

The Scenario

A well-meaning developer accidentally commits AWS credentials to a public GitHub repository. Automated scanners find it in minutes. Attackers start using the credentials before the developer even realizes the mistake.

Without secrets management: Credentials are valid until someone notices and rotates them (could be hours or days). Attackers spin up crypto miners, access S3 buckets, or worse.

With PAM/Secrets Management:

  • Developers retrieve credentials from vault at runtime, never storing them in code
  • Even if credentials leak, they rotate automatically (daily or more)
  • Leaked credentials are quickly invalid
  • Monitoring detects unusual access patterns (new IP, unusual API calls)

Ransomware Attacks

Modern ransomware doesn't just encrypt random files. Attackers specifically target privileged access to maximize damage.

The Modern Ransomware Playbook

1

Initial Compromise

Phishing email, RDP brute force, or exploited vulnerability gets attacker into one system.

2

Credential Harvesting

Attacker runs Mimikatz, searches for passwords in files, captures credentials from memory.

3

Domain Admin Achievement

Using harvested credentials, attacker escalates to Domain Admin. This is the key moment—with DA, they control everything.

4

Disable Defenses

With admin access, attacker disables antivirus, deletes backups, removes security tools.

5

Exfiltrate Data

Before encryption, attacker steals sensitive data for double extortion ("pay or we leak it").

6

Deploy Ransomware

Using domain admin access, attacker pushes ransomware to all systems simultaneously.

Where PAM Breaks the Kill Chain

Attack Stage PAM Defense
Credential Harvesting Admins don't type passwords (session proxy). No credentials on endpoints to harvest.
Pass-the-Hash lateral movement Unique passwords per system. Hash from one system doesn't work elsewhere.
Domain Admin escalation DA accounts have rotating, unknown passwords. JIT membership means no standing DA access.
Backup deletion Backup systems require PAM checkout. Unusual access triggers alerts.
Mass deployment Without domain admin credentials, attacker can't push to all systems.

Where KMS Protects Data

Attack Stage KMS Defense
Data exfiltration Data at rest is encrypted. Without KMS access, stolen database files are useless.
Double extortion If attacker can't decrypt the data, they have nothing to leak.
Backup theft Encrypted backups require KMS keys to restore. Stolen backups are unreadable.

Supply Chain Attacks

Sometimes attackers don't target you directly—they compromise something you trust.

Attack: Compromised Software Update

The Attack (SolarWinds Style)

Attackers compromise a software vendor's build process. Malicious code is inserted into a legitimate software update. When you install the "trusted" update, you install the malware. The malware often runs with high privileges because IT management software needs admin access.

How PAM/KMS Limit Damage:

  • Least privilege: Even if management software is compromised, it only has access to what it needs—not everything
  • Segmentation: Compromised software can't reach all systems if access is segmented
  • Monitoring: Unusual access patterns (management software querying unusual systems) trigger alerts
  • Credential isolation: Even if software is compromised, it doesn't have stored credentials to extract
  • Encrypted data: If attacker exfiltrates data, encryption renders it useless without keys

Attack: Third-Party Vendor Access

The Attack

You give a vendor VPN access for support. The vendor is breached. Attackers use the vendor's credentials to access your network.

With PAM for Vendor Access:

  • Just-in-time access: Vendor has no standing access—must request each time
  • Approval workflow: Your team approves each vendor access request
  • Session recording: Every vendor session is recorded and reviewable
  • Time limits: Access automatically expires after the support window
  • Scope limits: Vendor can only access specific systems they need

Mapping Defenses to Attacks

Here's a summary of which PAM/KMS controls defend against which attack techniques:

Attack Technique Key Defense How It Helps
Keyloggers Session proxy Passwords never typed by users
Memory scraping Session proxy + rotation Credentials not on endpoints; rotate after use
Hard-coded credentials Secrets management API No credentials in code/config
Pass-the-hash Unique passwords Hash only valid for one system
Kerberoasting Long random passwords Can't be cracked in useful time
SSH key theft Vaulted keys + certificates Keys not on endpoints; short-lived certs
Insider theft Session recording + monitoring All actions visible and auditable
Data exfiltration Encryption (KMS) Stolen data is unreadable
Ransomware Multiple PAM controls Blocks escalation path to domain admin
Vendor compromise JIT + approval + recording Limited scope, time, and full visibility

The Big Picture

Every sophisticated attack needs privileged credentials to succeed. PAM makes those credentials harder to steal (session proxy, no exposure), harder to use if stolen (unique passwords, rotation, MFA), and ensures attacks are detected (session recording, monitoring, SIEM integration). KMS ensures that even if attackers get the data, they can't read it. Together, they don't just raise the bar—they fundamentally change what's possible for attackers.
