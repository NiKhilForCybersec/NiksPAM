Company & Product Overview

CyberArk Software Ltd. was founded in 1999 in Israel and has grown to become the dominant PAM vendor globally. They pioneered the concept of the "Digital Vault" for securing privileged credentials.

Market Position

  • Gartner Magic Quadrant: Leader for 7+ consecutive years
  • Forrester Wave: Leader in PAM
  • Market share: Largest PAM vendor by revenue
  • Customer base: Over 8,000 organizations including 50%+ of Fortune 500

Self-Hosted vs. Privilege Cloud

Traditional CyberArk is self-hosted (you run the infrastructure). Privilege Cloud is the SaaS version where CyberArk manages the vault and components. Many organizations start self-hosted and migrate to cloud, or use hybrid models with cloud vault and on-prem connectors.

Network Architecture Considerations

  • Vault server: Place in most secure network zone (internal/restricted)
  • PVWA: Can be in DMZ or internal, depending on access requirements
  • PSM: Needs network access to target systems
  • CPM: Needs network access to target systems for password changes
  • Connectors: Bridge between components and target systems

Core Components

Digital Vault

The heart of CyberArk. A hardened server that stores all credentials encrypted.

  • Purpose: Secure storage of credentials, policies, and audit data
  • Technology: Proprietary hardened OS, not a standard Windows/Linux server
  • Encryption: AES-256 encryption, optional HSM integration for key protection
  • Access: Only vault protocol (port 1858), no RDP/SSH to vault
  • High Availability: Active-Passive clustering, DR replication

Vault Server Keys

The vault uses "Server Keys" to encrypt data. These keys are critical—losing them means losing access to all credentials. Keys can be stored on the vault server (default), on removable media, or in an HSM. For production, HSM is strongly recommended.

Password Vault Web Access (PVWA)

The web interface for users and administrators.

  • Purpose: Web portal for credential access, session launch, administration
  • Technology: IIS-based web application
  • Users: End users (access credentials), Administrators (manage platform)
  • Features: Dashboard, safe management, account management, reports, policies
  • Scalability: Multiple PVWA servers behind load balancer

Central Policy Manager (CPM)

Automated password management engine.

  • Purpose: Change passwords on target systems, verify passwords, reconcile mismatches
  • How it works: Connects to vault, retrieves accounts due for rotation, connects to targets, changes passwords
  • Platforms: Uses "platforms" (plugins) to know how to change passwords on different system types
  • Scalability: Multiple CPMs for distributed environments

CPM Operations

Operation What It Does When It Runs
Change Rotate password to new random value Scheduled or on-demand
Verify Confirm vault password matches target Scheduled or after change
Reconcile Reset password using privileged account when out of sync When verify fails

Privileged Session Manager (PSM)

Session proxy and recording server.

  • Purpose: Proxy privileged sessions, record all activity, enable monitoring
  • Technology: Windows Server with RDS (Remote Desktop Services)
  • Protocols: RDP, SSH, HTTPS (web apps), database clients, custom
  • Recording: Video recording stored in vault, searchable metadata
  • Monitoring: Live session view, ability to terminate sessions

PSM Connection Methods

  • PSM for Windows (RDP): Native RDP client connects to PSM, PSM proxies to target
  • PSM for SSH: SSH client connects to PSM, PSM proxies to target
  • PSM for Web: Browser-based access to web applications (via PSM)
  • PSMP (PSM Proxy): Linux-based SSH proxy (alternative to Windows PSM)

Privileged Threat Analytics (PTA)

Behavioral analytics and threat detection.

  • Purpose: Detect anomalous privileged activity, alert on threats
  • Data sources: Vault logs, Windows events, Syslog, network traffic
  • Detection: Machine learning baselines, rule-based alerts
  • Alerts: Suspected credential theft, unusual access patterns, risky commands

Deployment Models

Self-Hosted (On-Premises)

Traditional deployment where you manage all infrastructure:

Component Typical Sizing (Small) Typical Sizing (Large)
Vault 1 server + DR 1 server + DR per region
PVWA 2 servers (HA) 4+ servers behind LB
CPM 1-2 servers Multiple per region
PSM 2 servers Farm of 10+ servers
PTA 1 server Clustered

Privilege Cloud (SaaS)

CyberArk-managed vault in the cloud:

  • Vault: Managed by CyberArk in their cloud
  • PVWA: Cloud-hosted web interface
  • Connector: You deploy "Privilege Cloud Connector" on-premises
  • PSM: Can be cloud-hosted or on-premises
  • CPM: Runs on connector, manages on-prem systems

Privilege Cloud Benefits

Faster deployment (weeks vs. months), CyberArk handles patching/upgrades, built-in HA/DR, predictable subscription costs. Trade-off: Less control, data in CyberArk's cloud (may be concern for some industries).

Hybrid Deployment

Combination of cloud and on-premises:

  • Privilege Cloud vault + on-premises PSM servers
  • On-premises vault + cloud connectors for cloud workloads
  • Separate vaults for different security zones (IT vs. OT)

Safes & Policies

What is a Safe?

A "Safe" in CyberArk is a logical container for storing credentials. Think of it as a folder with access controls.

  • Purpose: Group related accounts, apply common permissions
  • Access control: Users/groups are granted permissions on safes
  • Typical organization: By department, application, environment, or sensitivity tier

Platforms (Policies)

A "Platform" defines how CyberArk manages a type of account:

  • Connection method: How PSM connects (RDP, SSH, etc.)
  • Password policy: Length, complexity, rotation frequency
  • Password change method: How CPM changes passwords on this system type
  • Verification method: How CPM verifies passwords

Built-in Platforms

  • Windows Server Local Accounts
  • Windows Domain Accounts
  • Unix via SSH
  • Oracle Database
  • SQL Server
  • Cisco IOS
  • VMware vCenter
  • AWS Access Keys
  • ...and many more

Custom Platforms

If CyberArk doesn't have a built-in platform for your system, you can create custom platforms. This involves writing scripts (PowerShell, shell) that CPM executes to change passwords. CyberArk Marketplace has community-contributed platforms.

Platform Management

Account Onboarding

Getting accounts into CyberArk:

1

Discovery

Use CyberArk Discovery to find privileged accounts (scans AD, servers, etc.)

2

Review

Pending accounts appear in discovery queue. Review and approve for onboarding.

3

Onboard

Assign to safe, select platform, optionally set initial password or let CPM change it.

4

Verify

CPM verifies it can connect with the credential.

5

Manage

Account is now managed—rotation, session recording, access control active.

Bulk Onboarding

For large environments, use:

  • Password Upload Utility: CSV-based bulk import
  • REST API: Programmatic account creation
  • Accounts Feed: Automatic onboarding from discovery

Linked Accounts

Some operations require multiple accounts. CyberArk supports "linked accounts":

  • Logon account: Account used to authenticate to target
  • Reconcile account: Privileged account to reset password if out of sync
  • Extra pass account: Additional accounts for complex scenarios

DevOps Integration (Conjur)

  • Ansible: Lookup plugin, credential injection
  • Jenkins: Credential binding plugin
  • Terraform: Provider for secrets retrieval

Key Takeaways

CyberArk is the market leader with the most comprehensive PAM platform. Core components: Vault (storage), PVWA (web UI), CPM (password management), PSM (session proxy), PTA (analytics).
