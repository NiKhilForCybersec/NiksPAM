Encryption Explained

Cryptography can seem intimidating, but the core concepts are surprisingly intuitive. This page explains encryption in plain language, with analogies that make the concepts stick.

What Is Encryption, Really?

At its core, encryption is scrambling information so only authorized parties can unscramble it. Think of it as a lock: the data is locked, and only someone with the key can unlock it.

The Fundamental Guarantee

Good encryption means: even if an attacker gets your encrypted data, they can't read it without the key. They could have the world's fastest computer and try for a billion years—they still won't crack it. This isn't an exaggeration; it's how modern cryptography works.

Key Terminology

Term Meaning Analogy
Plaintext The original, readable data A letter written in English
Ciphertext The encrypted, scrambled data The letter translated into an unreadable code
Key The secret that controls encryption/decryption The codebook that lets you translate
Encrypt Transform plaintext → ciphertext Lock the box
Decrypt Transform ciphertext → plaintext Unlock the box
Algorithm The mathematical method used The type of lock mechanism

Symmetric Encryption

Symmetric encryption uses the same key to encrypt and decrypt. It's called "symmetric" because both sides use identical keys.

The House Key Analogy

Symmetric encryption is like a house key. You lock the door with a key, and you unlock it with the same key. If you want someone else to enter, you need to give them a copy of your key. The challenge: how do you safely give them the key?

How It Works (Conceptually)

1

You Have Data

Your plaintext: "The secret meeting is at noon"

2

You Have a Key

A 256-bit random number (looks like: 7f3a9c2b...)

3

Encryption Happens

The algorithm (like AES) takes your data and key, performs mathematical operations, and outputs ciphertext: "x9Kp2mNv..."

4

Decryption Reverses It

With the same key, the algorithm reverses the operations, turning "x9Kp2mNv..." back into "The secret meeting is at noon"

The Pros and Cons

Pros Cons
Very fast - can encrypt gigabytes per second Key distribution problem - how do you share the key securely?
Efficient - low CPU usage Key management - need separate key for each pair of communicators
Well-understood - AES is heavily analyzed Both parties need the same key - if one is compromised, both are

Common Algorithms

  • AES (Advanced Encryption Standard): The gold standard. AES-256 is used everywhere.
  • ChaCha20: Modern alternative, often used in mobile (faster on devices without AES hardware)
  • 3DES: Legacy, being phased out—don't use for new systems

When It's Used

  • Encrypting files on disk
  • Database encryption
  • Encrypting the actual data in TLS (after keys are exchanged)
  • VPN tunnels

Asymmetric Encryption

Asymmetric encryption uses two different keys: a public key and a private key. What one encrypts, only the other can decrypt.

The Mailbox Analogy

Think of a mailbox with a slot. Anyone can drop a letter in (using the public key to encrypt). But only you have the key to open the mailbox and read the letters (using your private key to decrypt). You can give everyone your address (public key) without giving them access to your mail (private key).

The Key Pair

Key Who Has It What It Does
Public Key Everyone (it's public!) Encrypts data that only the private key can decrypt
Private Key Only you (keep it SECRET) Decrypts data that was encrypted with your public key

How It Solves Key Distribution

Remember symmetric encryption's problem—how do you share the key? Asymmetric solves this:

1

I Generate a Key Pair

I create a public key and private key. They're mathematically linked.

2

I Publish My Public Key

I can put this on my website, in a directory, email it—doesn't matter if attackers see it.

3

You Encrypt With My Public Key

You want to send me a secret? Encrypt it with my public key.

4

Only I Can Decrypt It

Even though everyone has my public key, only my private key can decrypt messages encrypted with it.

The Trade-off

Pros Cons
No key distribution problem - public keys are public Slow - 100-1000x slower than symmetric
Enables signatures - prove who sent something Larger keys - RSA needs 2048+ bits vs 256 for AES
Scalable - one key pair works with everyone Can only encrypt small amounts - data size limited

Common Algorithms

  • RSA: The classic. RSA-2048 or RSA-4096 for modern use.
  • Elliptic Curve (ECC): Modern, smaller keys, same security. P-256, P-384.
  • Ed25519: Popular for SSH keys and signatures.

Hashing Explained

Hashing is different from encryption. It's a one-way function: you can turn data into a hash, but you can't turn the hash back into data.

The Fingerprint Analogy

A hash is like a fingerprint for data. A fingerprint uniquely identifies a person, but you can't reconstruct the person from their fingerprint. Similarly, a hash uniquely identifies data, but you can't reconstruct the data from the hash.

Hash Properties

  • Deterministic: Same input always produces same output
  • One-way: Can't reverse the hash to get the original data
  • Fixed size: Output is always the same length (e.g., SHA-256 = 256 bits)
  • Avalanche effect: Tiny change in input = completely different output
  • Collision resistant: Practically impossible to find two inputs with the same hash

Example

Input SHA-256 Hash
"Hello" 185f8db32271fe2...9bd046c4ee15
"hello" 2cf24dba5fb0a30...e8b3c67898c
"Hello!" 33b59daa8a24e0f...1b2e6c4f2ae

Notice: Just changing a capital letter or adding one character produces a completely different hash.

What Hashing Is Used For

  • Password storage: Store hash of password, not the password itself. When user logs in, hash their input and compare.
  • Data integrity: Hash a file. Later, hash it again. If hashes match, file wasn't modified.
  • Digital signatures: Hash the document, then sign the hash (much faster than signing the whole document)
  • Deduplication: Two files have the same hash? They're identical.

Common Algorithms

  • SHA-256, SHA-384, SHA-512: The workhorses. Use SHA-256 for most purposes.
  • SHA-3: Newer standard, different design, equally secure.
  • MD5, SHA-1: BROKEN for security. Don't use for anything security-related.

Password Hashing Is Special

For passwords, don't use plain SHA-256. Use specialized password hashing functions (bcrypt, scrypt, Argon2) that are intentionally slow and use salt. This protects against brute force attacks. Regular hashes are too fast—attackers can try billions per second.

Digital Signatures

Digital signatures prove two things: who sent a message (authentication) and that the message wasn't tampered with (integrity).

The Wax Seal Analogy

In medieval times, important letters were sealed with wax imprinted with the sender's unique ring. The seal proved who sent it (only they have that ring) and that it wasn't opened (the seal would be broken). Digital signatures work the same way but are mathematically unforgeable.

How Digital Signatures Work

1

Hash the Message

Take your document/data and create a hash of it. This hash represents the document.

2

Encrypt Hash with Private Key

Use your private key to encrypt the hash. This encrypted hash IS the signature.

3

Send Document + Signature

Recipient gets your original document plus the signature.

4

Recipient Verifies

Recipient (1) hashes the document themselves, (2) decrypts your signature using your public key, (3) compares the two hashes. If they match, the signature is valid.

Why This Works

  • Authentication: Only you have your private key, so only you could create that signature
  • Integrity: If anyone changes the document, the hash changes, and the signature won't match
  • Non-repudiation: You can't deny signing it—your private key was required

Where Signatures Are Used

  • Software code signing (proving software came from legitimate developer)
  • Document signing (contracts, agreements)
  • SSL/TLS certificates (proving website identity)
  • Email signing (S/MIME, PGP)
  • Blockchain transactions

Envelope Encryption

Envelope encryption is the clever technique that combines the best of symmetric and asymmetric encryption. It's how most real-world systems work.

The Key in the Envelope Analogy

Imagine you want to send a large locked box (your data) to someone. You lock the box with a random key (data encryption key). Then you put that key inside a small envelope and lock the envelope with their padlock (their public key). They receive the box + envelope. They open the envelope with their key, get the box key, and open the box.

The Problem Envelope Encryption Solves

  • Symmetric is fast but has key distribution problems
  • Asymmetric solves key distribution but is slow and limited in data size
  • Envelope encryption: Use asymmetric for the small key, symmetric for the large data. Best of both worlds!

How It Works

1

Generate a Data Encryption Key (DEK)

Create a random symmetric key (e.g., AES-256) just for this data.

2

Encrypt the Data with DEK

Use the DEK to encrypt your actual data (fast, efficient).

3

Encrypt the DEK with Master Key

Use your master key (often in KMS/HSM) to encrypt the DEK. The DEK is small, so this is fast.

4

Store Encrypted Data + Encrypted DEK Together

The encrypted DEK is stored alongside (or with) the encrypted data.

To Decrypt

  1. Retrieve the encrypted DEK
  2. Use master key to decrypt the DEK
  3. Use the DEK to decrypt the data

Why This Is Brilliant

  • Master key stays protected: It only touches small DEKs, never bulk data
  • Easy rotation: To rotate master key, just re-encrypt the DEKs (don't re-encrypt all data)
  • Scalable: One master key can protect millions of DEKs
  • Fast: Bulk data encrypted with fast symmetric encryption

This is exactly how AWS KMS, Azure Key Vault, and most cloud encryption works.

How TLS Works

TLS (Transport Layer Security) is what puts the "S" in HTTPS. It's a real-world example showing how all these cryptographic pieces work together.

The TLS Handshake (Simplified)

1

Client Hello

Your browser says "Hi server, I want to connect securely. I support these encryption methods..."

2

Server Hello + Certificate

Server responds: "Great, let's use this encryption method. Here's my certificate (containing my public key) signed by a trusted authority."

3

Client Verifies Certificate

Browser checks: Is this certificate valid? Is it signed by a trusted CA? Does it match the domain? (This uses digital signatures)

4

Key Exchange

Browser and server use asymmetric cryptography (or Diffie-Hellman) to agree on a shared secret. Only they know this secret.

5

Derive Session Keys

Both sides use the shared secret to derive symmetric keys for this session.

6

Encrypted Communication

All data now flows encrypted with fast symmetric encryption (AES). An eavesdropper sees only gibberish.

What's Used Where

Cryptographic Primitive Role in TLS
Asymmetric (RSA/ECC) Server authentication (certificate), key exchange
Digital Signatures Certificate signing (proves certificate is legitimate)
Key Exchange (DH/ECDH) Derive shared secret without transmitting it
Symmetric (AES) Encrypt all actual data after handshake
Hashing (SHA-256) Message integrity (HMAC), key derivation

Real-World Applications

Let's connect the concepts to everyday security scenarios:

When You Log Into a Website

  1. TLS encrypts the connection (your password never travels in plaintext)
  2. Server receives your password
  3. Server hashes your password with a password hashing function
  4. Server compares hash to stored hash
  5. If match, you're authenticated

When PAM Stores a Password

  1. Password is encrypted with a Data Encryption Key (envelope encryption)
  2. DEK is encrypted with the Vault Key
  3. Vault Key is protected by HSM Master Key
  4. Master Key never leaves the HSM

When You Sign a PDF

  1. PDF software hashes the document
  2. Your private key encrypts the hash (creating signature)
  3. Signature is embedded in the PDF
  4. Anyone can verify using your public key

When Software Is Code-Signed

  1. Developer computes hash of the executable
  2. Developer signs hash with their code signing certificate's private key
  3. Signature is attached to the software
  4. When you run it, OS verifies signature proves it came from that developer and wasn't modified

Key Takeaways

Symmetric encryption (AES) is fast but has key distribution challenges. Asymmetric encryption (RSA, ECC) solves key distribution but is slow. Hashing is one-way and used for integrity and passwords. Digital signatures prove authenticity and integrity. Envelope encryption combines symmetric and asymmetric for the best of both. TLS shows how all these work together in practice.
