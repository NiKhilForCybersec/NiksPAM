Just-In-Time Access (JIT)

Just-In-Time access is a security model that eliminates standing privileges by granting access only when needed, for the minimum time required. This page explains JIT concepts, implementation approaches, and how it fits into Zero Trust architecture.

The JIT Concept

Just-In-Time (JIT) access is a privilege management approach where users receive access only when they need it, for a limited time, and with the minimum privileges required for their task. When the time expires or the task is complete, access is automatically revoked.

The Analogy: Hotel Room Keys

Think of a hotel room key. You get it when you check in, it works for your room only, and it expires when you check out. You don't get a permanent key to every room "just in case" you might need it someday. JIT access works the same way—you get exactly the access you need, exactly when you need it, and it goes away when you're done.

JIT Principles

Time-Bound

Access has a defined start and end time. When time expires, access is automatically revoked.

On-Demand

Users request access when they need it, rather than having it permanently assigned.

Least Privilege

Access is scoped to minimum necessary—specific systems, specific accounts, specific actions.

Justified

Every access request includes a reason/justification that's logged for audit.

The Problem with Standing Privileges

Standing privileges are access rights that remain assigned to a user continuously, whether or not they're actively using them. This is the traditional approach, and it creates significant risk.

Why Standing Privileges Are Dangerous

Risk Description
Expanded Attack Window If an attacker compromises a privileged account, those privileges are available 24/7. With JIT, they'd have to wait for the user to activate access.
Privilege Accumulation Over time, users accumulate access they no longer need. People change roles but keep old access. This violates least privilege.
Lateral Movement Attackers use stolen credentials to move laterally through the network. Standing privileges mean more paths are available.
Insider Threat Disgruntled employees have constant access to sensitive systems. With JIT, their access window is limited.
Compliance Burden Auditors ask who has access and why. With standing privileges, the answer is often unclear or outdated.

Real-World Impact

In many breaches, attackers gain access to an account and find it has Domain Admin privileges that were granted years ago for a one-time project. With JIT, that Domain Admin access wouldn't exist unless actively requested and approved for a current need.

Standing vs JIT Comparison

Aspect Standing Privileges JIT Access
When access exists Always (24/7/365) Only during approved window
Attack surface Maximum—all privileges always exposed Minimal—privileges only exist when active
Audit complexity Must justify all existing access Every access has built-in justification
User friction None—access always available Some—must request access each time
Compromised credential risk High—attacker inherits all privileges Lower—attacker must wait for activation

How JIT Works

A typical JIT workflow involves these stages:

1

Access Request

User submits a request for specific access: which system, which account/role, how long needed, and why (business justification). This may be manual or automated (e.g., triggered by a ticket).

2

Policy Evaluation

PAM evaluates the request against policies: Is this user eligible for this access? Is the justification valid? Does this require approval, or can it be auto-approved?

3

Approval (If Required)

For sensitive access, the request routes to an approver (manager, security team, system owner). They review and approve or deny the request.

4

Access Provisioning

Upon approval, access is granted. This might mean: adding user to a group, enabling an account, granting a role, or creating temporary credentials.

5

User Performs Work

User has access for the approved duration. Session may be recorded. Activity is logged.

6

Automatic Revocation

When time expires (or user signals they're done), access is automatically revoked. User is removed from groups, account is disabled, credentials are rotated.

7

Audit Trail

Complete record exists: who requested, why, who approved, when access was granted, what was done, when access was revoked.

Implementation Approaches

There are several technical approaches to implementing JIT access:

1. JIT Group Membership

User is temporarily added to a privileged group (e.g., Domain Admins) for the duration of the request.

How It Works
  • User normally has no membership in privileged AD groups
  • Upon approval, PAM adds user to the group
  • User's token/session refreshes to include new group
  • When time expires, PAM removes user from group

Pros: Works with existing infrastructure, no application changes needed.

Cons: Token refresh required (may need re-login), coarse-grained (whole group membership).

2. JIT Account Activation

Privileged accounts exist but are disabled by default. JIT enables the account temporarily.

How It Works
  • Dedicated privileged account per user (e.g., admin-jsmith) exists but is disabled
  • Upon approval, PAM enables the account and provides the password
  • User logs in with the enabled account
  • When time expires, PAM disables the account and rotates the password

Pros: Clear separation between regular and privileged identity.

Cons: More accounts to manage, account provisioning overhead.

3. JIT Role Assignment (Cloud)

For cloud platforms, temporarily assign IAM roles to users.

How It Works
  • User has no standing IAM policies/roles for sensitive resources
  • Upon approval, PAM assigns the required role (e.g., AWS IAM Role)
  • User assumes the role for their session
  • When time expires, role assignment is removed

Pros: Native cloud integration, fine-grained permissions.

Cons: Cloud-specific, may require different approach per cloud.

4. Dynamic/Ephemeral Credentials

Generate short-lived credentials on demand that automatically expire.

How It Works
  • No permanent credentials exist
  • Upon request, system generates temporary credentials (certificate, token, password)
  • Credentials have built-in expiration (e.g., 1 hour TTL)
  • No revocation needed—credentials simply stop working

Pros: Most secure—no standing credentials anywhere. HashiCorp Vault excels at this.

Cons: Requires application/infrastructure changes to support dynamic credentials.

Zero Standing Privileges (ZSP)

Zero Standing Privileges (ZSP) is the ultimate goal of JIT access—a state where NO user has any standing privileged access. All privileged access is JIT.

What ZSP Means

Zero Standing Privileges Defined
  • No user is a permanent member of privileged groups (Domain Admins, etc.)
  • No privileged accounts are permanently enabled
  • No standing IAM policies grant sensitive permissions
  • All privileged access flows through JIT workflows
  • Even "break-glass" emergency access is managed (just with expedited workflow)

ZSP Maturity Journey

1

Level 1: Vaulting

Privileged credentials are stored in PAM vault. Users still have standing access, but passwords are managed.

2

Level 2: JIT for High-Risk

JIT implemented for the most sensitive access (Domain Admin, production databases). Other privileges remain standing.

3

Level 3: JIT as Default

Most privileged access is JIT. Standing privileges only for exceptional cases with strong justification.

4

Level 4: Zero Standing Privileges

No standing privileged access anywhere. 100% JIT. Even emergency access is JIT with expedited approval.

ZSP Is a Journey

Most organizations don't achieve ZSP overnight. It requires significant process changes, user training, and cultural shift. Start with the highest-risk access and expand gradually. The goal is continuous reduction of standing privileges.

JIT and Zero Trust

JIT access is a fundamental component of Zero Trust architecture. Zero Trust assumes no implicit trust—every access request must be verified.

Zero Trust Principles and JIT

Zero Trust Principle How JIT Supports It
Never Trust, Always Verify Every JIT request is verified against policy, even for "trusted" users
Least Privilege JIT grants minimum required access for minimum required time
Assume Breach JIT limits blast radius—compromised account has no standing privileges
Continuous Verification Each JIT request is a new verification point, not just initial login
Microsegmentation JIT enables fine-grained access to specific resources, not broad network access

JIT + Zero Trust = Defense in Depth

In a Zero Trust model, even if an attacker bypasses perimeter controls and compromises a user account, they face another barrier: the user has no standing privileges. They'd have to also compromise the JIT workflow to gain privileged access.

Best Practices

Implementation Best Practices

Getting JIT Right
  • Start with highest-risk access: Domain Admin, database admin, cloud admin roles
  • Define reasonable time windows: Not too short (frustrating) or too long (defeats purpose). 4-8 hours typical.
  • Allow extensions: Users should be able to request more time before expiration, with justification
  • Automate revocation: Never rely on manual cleanup—access must expire automatically
  • Include emergency procedures: Define break-glass process for urgent access when approvers unavailable

User Experience Best Practices

Minimizing Friction
  • Self-service portal: Users should be able to request access without opening tickets
  • Fast approval: Mobile approval, Slack/Teams integration for quick turnaround
  • Pre-approved access: For routine tasks, define pre-approved JIT that doesn't require manager sign-off
  • Remember preferences: Save common request patterns for quick re-request
  • Clear status: Users should easily see what access they have and when it expires

Security Best Practices

Maintaining Security
  • Require MFA for JIT requests: Extra verification before granting privileged access
  • Log everything: Request, approval, provisioning, activity, revocation
  • Review patterns: Regularly review JIT usage for anomalies or abuse
  • Limit approver scope: Approvers should only approve access they understand
  • Test revocation: Regularly verify that access is actually revoked when it should be

Key Takeaways

JIT access eliminates standing privileges by granting access only when needed, for limited time, with automatic revocation. It's a core component of Zero Trust architecture and significantly reduces the attack surface. Start with high-risk access and expand gradually toward Zero Standing Privileges.
