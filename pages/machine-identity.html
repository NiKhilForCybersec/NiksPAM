Machine Identity Management

Machine identities—certificates, API keys, service accounts, and tokens—now outnumber human identities by 45:1 in most enterprises. Managing them is critical for security.

What is Machine Identity?

A machine identity is a credential that identifies and authenticates a non-human entity—a server, application, container, IoT device, or automated process. Just as humans need usernames and passwords, machines need identities to communicate securely.

The Scale Problem

Era Human:Machine Ratio Characteristic
Traditional IT 1:1 One server, one service account
Virtualization 1:10 Multiple VMs per physical server
Cloud/Containers 1:45+ Ephemeral workloads, microservices, serverless
IoT 1:1000+ Thousands of devices per deployment

The Neglected Attack Surface

Organizations invest heavily in human identity management (SSO, MFA, PAM) but often neglect machine identities. Expired certificates cause outages. Stolen API keys enable breaches. Orphaned service accounts become backdoors. Machine identity management is as critical as human IAM.

Types of Machine Identities

Type What It Is Example Use Cases
TLS/SSL Certificates X.509 certificates for encryption and server authentication HTTPS websites, service-to-service TLS, mTLS
Code Signing Certificates Certificates that prove software authenticity Software distribution, driver signing, container images
SSH Keys Public/private key pairs for SSH authentication Server access, Git operations, automation
API Keys Static strings for API authentication Third-party services, internal APIs, cloud services
OAuth Tokens Time-limited access tokens Application authorization, API access
Service Accounts Non-human accounts in identity systems AD service accounts, cloud IAM principals
Managed Identities Cloud-native workload identities Azure Managed Identity, AWS IAM roles for EC2
SPIFFE/SPIRE Workload identity standard Service mesh identity, zero trust workloads

Certificate Management

Certificates are the most common and complex machine identity to manage.

Certificate Lifecycle

1

Request

Application or system needs a certificate. Generate key pair, create Certificate Signing Request (CSR).

2

Issuance

CA validates request, signs certificate. Public CA for external, private CA for internal.

3

Deployment

Certificate and private key installed on server/application. Key must be protected.

4

Monitoring

Track expiration dates, usage, compliance. Alert before expiration.

5

Renewal

Before expiration, renew certificate. Automated renewal prevents outages.

6

Revocation

If compromised or no longer needed, revoke immediately. Publish to CRL/OCSP.

Certificate Management Challenges

  • Inventory: Where are all certificates? Many organizations don't know.
  • Expiration: Certificates expire. Expired certs = outages. (Remember the 2020 Microsoft Teams outage?)
  • Key protection: Private keys must be secured. Stolen key = impersonation.
  • Sprawl: DevOps creates certificates on demand. Shadow PKI emerges.
  • Compliance: Key length requirements, algorithm deprecation (SHA-1, RSA-1024)

Automated Certificate Management (ACME)

ACME protocol (used by Let's Encrypt) enables automated certificate issuance: 

# Using certbot for Let's Encrypt
certbot certonly --webroot -w /var/www/html \
    -d example.com -d www.example.com

# Automatic renewal (cron or systemd timer)
certbot renew --quiet

Private CA for Internal Services

For internal service-to-service TLS, run your own CA:

  • HashiCorp Vault PKI: Dynamic, short-lived certificates
  • AWS Private CA: Managed private CA in AWS
  • Azure Key Vault: Certificate issuance and management
  • Smallstep: Open-source private CA with ACME support
  • CFSSL: CloudFlare's PKI toolkit
# HashiCorp Vault PKI - Issue certificate with 24h TTL
vault write pki/issue/my-role \
    common_name="myservice.internal" \
    ttl="24h"

# Returns certificate, private key, and CA chain

API Keys & Tokens

API Key Risks

  • No expiration: Many API keys never expire
  • Over-privileged: Keys often have more access than needed
  • Scattered: Keys in config files, environment variables, code
  • Shared: Same key used across multiple applications
  • Untracked: No inventory of which keys exist

API Key Best Practices

Practice Why
Unique keys per application Revoke one without affecting others
Least privilege Key only has permissions it needs
Expiration Force rotation, limit breach window
Secure storage In secrets manager, not code/config
Rotation Regular rotation even without breach
Monitoring Detect unusual usage patterns

Short-Lived Tokens (Better Than Keys)

Modern systems use tokens instead of static keys:

# OAuth 2.0 Client Credentials Flow
POST /oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=my-app
&client_secret=secret123
&scope=api:read

# Response
{
  "access_token": "eyJhbGciOiJSUzI1...",
  "token_type": "Bearer",
  "expires_in": 3600  // 1 hour
}

Service Accounts

Active Directory Service Accounts

Traditional AD service accounts have many problems:

  • Passwords rarely rotated (breaking things is scary)
  • Over-privileged (Domain Admins "because it was easier")
  • Orphaned (person who created it left, nobody knows what it does)
  • Shared (same account for multiple applications)
  • Exposed to Kerberoasting attacks

Group Managed Service Accounts (gMSA)

Microsoft's solution for Windows service accounts:

  • AD-managed password: 240-character password, automatically rotated every 30 days
  • No human knows password: AD and authorized servers retrieve it automatically
  • No Kerberoasting: Password too complex to crack
  • Single account, multiple servers: Can be used across server farm
# Create gMSA in PowerShell
New-ADServiceAccount -Name "svc_webapp" `
    -DNSHostName "svc_webapp.corp.example.com" `
    -PrincipalsAllowedToRetrieveManagedPassword "WebServers"

# Install on server
Install-ADServiceAccount -Identity "svc_webapp"

# Configure Windows service to use gMSA
sc.exe config MyService obj= "CORP\svc_webapp$"

Cloud Service Accounts

Cloud Service Account Type Best Practice
AWS IAM Users (avoid) or IAM Roles (preferred) Use roles with temporary credentials, not access keys
Azure Service Principals or Managed Identities Use managed identities where possible (no credentials to manage)
GCP Service Accounts Use Workload Identity Federation, avoid key files

Workload Identity

Modern workload identity systems provide identities to applications without static credentials.

Cloud Workload Identity

AWS: IAM Roles for EC2/ECS/Lambda

# EC2 instance gets credentials automatically via metadata service
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/my-role

# Returns temporary credentials
{
  "AccessKeyId": "ASIA...",
  "SecretAccessKey": "...",
  "Token": "...",
  "Expiration": "2024-01-01T12:00:00Z"
}

Azure: Managed Identity

# Azure VM/Container gets token from metadata service
curl 'http://169.254.169.254/metadata/identity/oauth2/token?
    api-version=2018-02-01&resource=https://vault.azure.net' \
    -H Metadata:true

# Use token to access Azure resources - no credentials stored!

SPIFFE/SPIRE

SPIFFE (Secure Production Identity Framework For Everyone) is an open standard for workload identity:

  • SPIFFE ID: URI-based identity (spiffe://example.com/service/frontend)
  • SVID: Short-lived X.509 certificate or JWT proving identity
  • SPIRE: Reference implementation of SPIFFE
# SPIFFE identity format
spiffe://trust-domain/path/to/workload

# Examples
spiffe://example.com/ns/production/sa/frontend
spiffe://example.com/cluster/k8s-west/ns/default/pod/web-abc123

Kubernetes Workload Identity

Connect Kubernetes service accounts to cloud IAM:

# AWS: EKS Pod Identity / IRSA
# Associates K8s service account with AWS IAM role

apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/my-app-role

---
# Azure: Workload Identity
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
  annotations:
    azure.workload.identity/client-id: "00000000-0000-0000-0000-000000000000"

Lifecycle Management

Discovery

First, find all machine identities:

  • Certificate scanning: Scan networks for TLS endpoints, extract certificates
  • Cloud API enumeration: List service accounts, roles, keys via cloud APIs
  • Code scanning: Search repos for hardcoded credentials, API keys
  • AD enumeration: Query for service accounts, SPNs
  • Secrets manager inventory: Export list of stored secrets

Inventory & Classification

For each machine identity, track:

  • What is it? (Certificate, API key, service account, etc.)
  • Where is it used? (Which servers, applications, services)
  • Who owns it? (Team, individual responsible)
  • What can it access? (Permissions, scope)
  • When does it expire? (If applicable)
  • When was it last used? (Detect orphaned identities)

Rotation

Identity Type Recommended Rotation Approach
TLS Certificates 90 days (or shorter) Automated via ACME or PKI
Code Signing Certs 1-3 years Manual with HSM protection
SSH Keys 90 days or use certificates SSH certificate authority preferred
API Keys 90 days Automated rotation in secrets manager
Service Account Passwords 30-90 days or use gMSA PAM automated rotation or gMSA
Cloud Access Keys Avoid entirely Use workload identity instead

Revocation

When machine identity is compromised or no longer needed:

  • Certificates: Revoke via CA, publish to CRL/OCSP
  • API keys: Delete from provider, rotate all copies
  • Service accounts: Disable account, rotate password, then delete
  • Cloud roles: Remove role assignments, delete role if not shared

Key Takeaways

Machine identities outnumber humans 45:1 and are growing. They include certificates, API keys, service accounts, and tokens. Certificate management is critical—expired certs cause outages. Prefer short-lived, automatically rotated credentials. Use workload identity (cloud-native, SPIFFE) over static keys. Maintain inventory, track ownership, and monitor usage. Machine identity management is as important as human IAM.
