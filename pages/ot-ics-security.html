PAM for OT/ICS Environments

Operational Technology (OT) and Industrial Control Systems (ICS) present unique challenges for privileged access management. This guide covers how to secure privileged access in manufacturing, utilities, energy, and other industrial environments.

OT vs. IT: Key Differences

Before implementing PAM in OT environments, you must understand how OT differs from traditional IT:

Aspect IT Environment OT Environment
Priority Confidentiality → Integrity → Availability (CIA) Safety → Availability → Integrity → Confidentiality
Downtime tolerance Planned maintenance windows acceptable 24/7/365 operation often required
System lifespan 3-5 years typical 15-30 years common
Patching Regular patching expected Patching may be impossible or rare
Change management Agile, frequent changes Highly controlled, infrequent changes
Real-time requirements Generally none Millisecond timing may be critical
Physical safety Rarely a concern Can cause injury, death, or environmental damage
Vendor dependency Multiple vendors, competitive market Often single-vendor systems, proprietary

Safety First

In OT, security controls must NEVER compromise safety. A PAM system that blocks emergency access to a safety system could cause injury or death. Always work with operations teams to understand safety implications before implementing controls.

OT System Types

  • SCADA (Supervisory Control and Data Acquisition): Monitors and controls distributed systems (pipelines, power grids)
  • DCS (Distributed Control Systems): Controls industrial processes (refineries, chemical plants)
  • PLC (Programmable Logic Controllers): Automated control of machinery
  • HMI (Human Machine Interface): Operator workstations for monitoring/control
  • RTU (Remote Terminal Units): Field devices that collect data and execute commands
  • SIS (Safety Instrumented Systems): Emergency shutdown systems—highest criticality

PAM Placement in Purdue Model

  • Level 3.5 (Industrial DMZ): PAM jump servers and session proxies placed here
  • Never direct access: Users from Level 4/5 should never connect directly to Level 2/1/0
  • PAM as gateway: All access to lower levels flows through PAM in the DMZ
  • Vault location: Can be in IT network with secure communication to OT PAM components

OT-Specific Challenges

1. Legacy Systems

Many OT systems run outdated software that can't be easily updated:

  • Windows XP/2003 still common in industrial environments
  • Proprietary operating systems with no modern authentication
  • Hardcoded credentials that can't be changed
  • Systems with no encryption capability

Legacy Workaround

For systems that can't rotate passwords or support modern auth, PAM can still add value: control access to jump servers, record all sessions to the legacy system, and provide an audit trail even if you can't change the underlying credential.

2. Availability Requirements

  • Can't implement controls that might cause outages
  • Password rotation during production could be catastrophic
  • MFA failures can't block emergency access
  • Network latency for PAM can't impact real-time control

3. Air-Gapped Networks

Some OT networks have no connection to IT or internet:

  • PAM must operate standalone (no cloud dependencies)
  • Updates and patches delivered manually
  • No SIEM integration (or one-way data diode)
  • Separate identity infrastructure

4. Vendor Maintenance

  • Equipment vendors need remote access for maintenance
  • Often use shared credentials ("vendor password")
  • May require broad access with minimal oversight
  • Contractual obligations may limit security controls

5. Protocol Diversity

OT uses many protocols that IT PAM solutions don't natively support:

  • Modbus (serial and TCP)
  • DNP3 (Distributed Network Protocol)
  • OPC UA/DA
  • IEC 61850, IEC 60870-5-104
  • Proprietary vendor protocols

Architecture Options

Model Description When to Use
Centralized Vault Single vault in IT, connectors in OT DMZ Connected environments, single site
Distributed Vault Separate vault in OT network, replicated or independent Air-gapped networks, multiple sites
Hybrid IT vault for IT accounts, OT vault for OT accounts Different compliance requirements
Cloud + On-Prem Cloud PAM for IT, on-prem for OT IT modernizing, OT staying traditional

Air-Gapped Deployment

For truly isolated OT networks:

  • Standalone PAM: Complete PAM infrastructure within OT network
  • Local identity: Separate AD or local PAM users (no IT AD)
  • Manual updates: PAM updates via secure media transfer
  • Data diode: One-way flow of audit data to IT SIEM (optional)
  • No internet: No license checks, cloud features, external MFA

Vendor Remote Access

Equipment vendors are a major attack vector in OT. Third-party vendors are often compromised and used to access customer networks. PAM is critical for controlling vendor access.

Vendor Access Challenges

  • Vendors need periodic access for maintenance and troubleshooting
  • Often demand "always-on" VPN access
  • May resist session recording (claim proprietary methods)
  • Use shared credentials ("Siemens support account")
  • Multiple vendor employees, one "vendor" account

PAM Solution for Vendor Access

1

Individual Vendor Accounts

Each vendor employee gets their own PAM account ([email protected]). No shared "vendor" accounts.

2

Just-In-Time Access

No standing access. Vendor requests access for specific work window. Your team approves each request.

3

Scoped Access

Vendor only gets access to their specific equipment. Siemens can't see ABB systems and vice versa.

4

Full Session Recording

Every keystroke, every screen recorded. Can be reviewed later. Provides accountability.

5

Automatic Expiration

Access expires after approved window. No "permanent" vendor access.

Vendor Pushback

Vendors may resist individual accounts and session recording. Stand firm. It's your network, your risk. Professional vendors understand and accept security requirements. If a vendor refuses basic security controls, that's a red flag about their own security practices.

VPN-less Vendor Access

Modern approach eliminates traditional VPN:

  • Zero Trust Network Access (ZTNA): Vendor accesses specific application, not full network
  • PAM Web Portal: Browser-based access, no VPN client needed
  • No network exposure: Vendor can't scan or pivot
  • Works from anywhere: Vendor doesn't need corporate network

Industrial Protocols

PAM session recording works well for standard protocols (RDP, SSH, VNC). Industrial protocols are more challenging:

Protocol Coverage

Protocol PAM Approach Considerations
RDP to HMI Full PSM recording Standard PAM use case
SSH to Linux servers Full PSM recording Standard PAM use case
VNC PSM recording (many PAM vendors support) Common in OT for legacy systems
Telnet PSM proxies and records Insecure but still used; PAM makes it safer
Vendor engineering tools Record the workstation session, not the protocol User connects to jump server via PSM, runs vendor tool
Serial connections Serial-to-IP converters + PAM Convert to network protocol, then proxy
Modbus/DNP3 Network monitoring, not PAM recording OT-specific security tools (Claroty, Nozomi)

Engineering Workstation Approach

For proprietary vendor protocols, use an engineering workstation model:

  1. Engineer connects via PAM to jump server (RDP, recorded)
  2. Jump server has vendor software installed (Step7, FactoryTalk, etc.)
  3. Engineer uses vendor software to connect to equipment
  4. The RDP session captures everything they do
  5. Network monitoring tools can capture the protocol traffic separately

OT Compliance Frameworks

NERC CIP (North American Energy)

Critical Infrastructure Protection standards for the bulk electric system:

Standard Requirement PAM Coverage
CIP-004-6 Personnel & Training Access management, training records, access reviews
CIP-005-6 Electronic Security Perimeter PAM enforces ESP access controls, remote access management
CIP-006-6 Physical Security Logical access to physical security systems
CIP-007-6 System Security Management Password management, access control enforcement
CIP-011-2 Information Protection Encryption via KMS, access logging

IEC 62443 (Industrial Automation)

International standard for industrial cybersecurity:

Requirement PAM Coverage
IAC (Identification and Authentication Control) Strong authentication, unique user IDs
UC (Use Control) Authorization enforcement, least privilege, session limits
SI (System Integrity) Audit logging, tamper detection
DC (Data Confidentiality) Credential encryption, session encryption via KMS
TRE (Timely Response to Events) Real-time monitoring, alerting, session termination

Industry-Specific Standards

  • Pharmaceutical (21 CFR Part 11): Electronic records, electronic signatures, audit trails
  • Oil & Gas (API): Pipeline security, SCADA protection
  • Water (AWIA): America's Water Infrastructure Act security requirements
  • Transportation (TSA): Pipeline and rail cybersecurity directives

Implementation Approach

Phase 1: Assessment & Planning (Months 1-2)

  • Inventory all OT systems and privileged accounts
  • Map network architecture (Purdue levels)
  • Identify vendor access requirements
  • Assess which systems can have passwords rotated
  • Identify safety-critical systems requiring special handling
  • Engage operations teams early—their buy-in is critical

Phase 2: Infrastructure (Months 2-3)

  • Deploy PAM infrastructure in Industrial DMZ
  • Configure network segmentation (if not already in place)
  • Set up jump servers with required vendor tools
  • Test connectivity to OT systems
  • Establish break-glass procedures for emergencies

Phase 3: Pilot (Months 3-4)

  • Start with Level 3 systems (historians, engineering workstations)
  • Onboard a small group of users and accounts
  • Implement session recording without blocking access
  • Test password rotation on non-critical systems only
  • Gather feedback, adjust policies

Phase 4: Expansion (Months 4-6)

  • Extend to Level 2 systems (HMI, SCADA servers)
  • Implement vendor access controls
  • Enable approval workflows for critical systems
  • Begin password rotation where safe to do so

Phase 5: Hardening (Months 6+)

  • Remove direct access paths (force through PAM)
  • Implement real-time monitoring and alerting
  • Regular access reviews
  • Continuous improvement based on operational experience

Keys to OT PAM Success

Work with operations: They know the systems and risks. Don't break production: Implement controls gradually. Plan for emergencies: Break-glass must work when needed. Accept limitations: Some legacy systems can't be fully managed—do what you can. Focus on vendors: Often the biggest risk and quickest win.
