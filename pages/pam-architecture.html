PAM Architecture Patterns

PAM solutions can be deployed in various configurations depending on your environment, security requirements, and scale. This page covers common deployment models, network architectures, and integration patterns.

Deployment Models

PAM solutions are available in several deployment models, each with different trade-offs for control, complexity, and cost.

On-Premises

On-Premises Deployment
  • Description: PAM software deployed on your own infrastructure
  • Examples: CyberArk Self-Hosted, BeyondTrust on-prem, Delinea Secret Server
  • Pros: Full control, data stays on-premises, can operate air-gapped
  • Cons: Infrastructure costs, maintenance burden, longer deployment time
  • Best for: Highly regulated industries, government, air-gapped environments

SaaS / Cloud-Hosted

SaaS Deployment
  • Description: Vendor operates PAM in their cloud, you access via web
  • Examples: CyberArk Privilege Cloud, Delinea Secret Server Cloud, 1Password Business
  • Pros: Fastest deployment, no infrastructure to manage, automatic updates
  • Cons: Data in vendor cloud, depends on internet connectivity, less customization
  • Best for: Cloud-first organizations, faster time to value, smaller teams

Private Cloud / IaaS

Private Cloud Deployment
  • Description: Deploy PAM on cloud infrastructure you control (AWS, Azure, GCP)
  • Examples: CyberArk on AWS/Azure, HashiCorp Vault on Kubernetes
  • Pros: Cloud elasticity, you control the environment, can be isolated in VPC
  • Cons: You manage infrastructure, cloud costs, networking complexity
  • Best for: Cloud-heavy organizations wanting control, hybrid environments

Hybrid

Hybrid Deployment
  • Description: Combination of on-premises and cloud components
  • Examples: SaaS vault with on-premises connectors/jump servers
  • Pros: Best of both worlds, reach on-premises targets from cloud platform
  • Cons: Most complex, multiple components to manage
  • Best for: Organizations with significant on-premises infrastructure moving to cloud

Principle: Users Never Connect Directly to Targets

In a proper PAM architecture, users never have direct network access to privileged systems. All access flows through PAM. This may require network changes—blocking direct RDP/SSH from user networks to server networks.

High Availability

PAM is a critical security control. If PAM is unavailable, administrators can't access systems (except via break-glass). High availability is essential.

HA Components

Component HA Approach Notes
Vault / Database Active/Passive cluster, database replication Most critical—contains all secrets
Web Portal Multiple instances behind load balancer Stateless, easy to scale
Session Proxy (PSM) Multiple instances, load balanced Scale based on concurrent sessions
Password Manager (CPM) Multiple instances, work distribution Active/Active for rotation tasks

Disaster Recovery

DR Considerations
  • Vault replication: Replicate vault data to DR site (sync or async)
  • RTO/RPO: Define recovery time and recovery point objectives
  • DR site components: Full PAM stack at DR site or minimal (vault only)
  • DNS failover: Update DNS to point to DR site during failover
  • Regular testing: Test DR failover at least annually

Connector/Gateway

Connector/Gateway Pattern

Most PAM solutions use connectors or gateways deployed in each environment to:

  • Proxy sessions to local targets (avoiding hairpinning traffic through central site)
  • Perform password rotation against local systems
  • Discover local privileged accounts
  • Communicate securely back to central vault

Cloud-Native Considerations

  • Cloud IAM: Integrate with AWS IAM, Azure AD, GCP IAM for cloud resource access
  • Managed services: Some cloud resources don't support traditional credential management
  • Dynamic infrastructure: Auto-scaling instances need dynamic PAM enrollment
  • Containers/Kubernetes: Secrets injection for containerized workloads

Tiered Architecture (AD Tiering)

Microsoft's Administrative Tier Model segments privileged access to prevent credential theft from propagating across the environment. PAM should align with this model.

The Three Tiers

Tier Assets Example Accounts PAM Treatment
Tier 0 Domain Controllers, AD, PKI, PAM itself Domain Admins, Enterprise Admins Highest protection, JIT only, multi-person approval
Tier 1 Servers, applications, databases Server admins, DBA, app admins Vaulted credentials, session recording
Tier 2 Workstations, end-user devices Help desk, local admin LAPS or vaulted, may be less strict

Key Principles

Tier Isolation Rules
  • No downward authentication: Tier 0 accounts never log into Tier 1 or Tier 2 systems
  • No upward access: Tier 2 systems can't access Tier 0 or Tier 1
  • Separate admin accounts: Different accounts for each tier
  • Privileged Access Workstations (PAWs): Dedicated hardened workstations for Tier 0 administration

PAM Enables Tiering

Without PAM, enforcing tiering is very difficult. Users tend to use the same admin account everywhere. PAM enforces tiering by: providing different credentials for different tiers, routing sessions through appropriate jump servers, and detecting cross-tier credential usage.

SIEM Integration Pattern

1

Configure Log Forwarding

PAM sends events via Syslog (UDP/TCP) or API to SIEM.

2

Parse and Normalize

SIEM parses PAM logs into structured fields (user, action, target, result).

3

Create Correlation Rules

Build detection rules: failed auth attempts, after-hours access, credential theft indicators.

4

Build Dashboards

Visualize PAM activity: session counts, top users, credential checkouts, alerts.

CI/CD Integration Pattern

1

Pipeline Authenticates to PAM

CI/CD tool authenticates using service account, certificate, or cloud identity.

2

Request Secrets

Pipeline calls PAM API to retrieve deployment credentials (DB passwords, API keys).

3

Use Secrets

Credentials used for deployment, then discarded (not stored in pipeline config).

4

Audit Trail

PAM logs which pipeline retrieved which secrets, enabling troubleshooting and security review.

Sizing Considerations

PAM infrastructure should be sized based on your environment's scale and usage patterns.

Key Sizing Factors

Factor Impact Component Affected
Number of accounts Vault storage, rotation load Vault, CPM
Concurrent sessions Session proxy capacity PSM
Session recording Storage for recordings Storage, PSM
Rotation frequency CPM workload CPM
API request volume Web server capacity Web portal, API server
Geographic distribution Number of connector sites Connectors, network

Typical Sizing Guidelines

General Guidelines

Small: <5,000 accounts, <50 concurrent sessions — Single vault, 1-2 PSMs, 1-2 CPMs.
Medium: 5,000-50,000 accounts, 50-200 sessions — Clustered vault, 2-4 PSMs, 2-4 CPMs.
Large: >50,000 accounts, 200+ sessions — Multi-site deployment, scaled PSM/CPM, dedicated recording storage.

Key Takeaways

PAM architecture depends on your deployment model (on-prem, SaaS, hybrid), network design (proper segmentation, no direct user-to-target access), high availability requirements, and integrations. Align PAM with tiered administration for Windows environments. Size appropriately for your account count and session volume.
