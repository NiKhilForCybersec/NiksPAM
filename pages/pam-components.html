PAM Core Components

A PAM solution is made up of several interconnected components, each serving a specific purpose. Understanding these components helps you design, implement, and troubleshoot PAM deployments effectively.

Vendor Terminology

Throughout this page, we'll use generic terms. Here's how they map to common vendors: Vault = CyberArk Digital Vault, Delinea Secret Server, BeyondTrust Password Safe. Session Manager = CyberArk PSM, BeyondTrust Session Manager, Delinea Connection Manager. Password Manager = CyberArk CPM, auto-rotation engines in other products.

Credential Vault

The Credential Vault (also called Digital Vault, Password Vault, or Secret Server) is the heart of any PAM solution. It's a hardened, encrypted repository that stores all privileged credentials.

What It Does

Encrypted Storage

Stores credentials encrypted at rest using strong algorithms (typically AES-256). Even database administrators can't read the passwords.

Access Control

Enforces fine-grained policies determining who can access which credentials, when, and under what conditions.

Audit Logging

Records every access attempt, successful or failed, with timestamps, user identity, and source IP.

High Availability

Typically deployed in clustered or replicated configuration to ensure credentials are always accessible.

Key Technical Details

Aspect Description
Encryption AES-256 for data at rest; TLS 1.2/1.3 for data in transit. Master key often protected by HSM.
Database Backend Usually proprietary format or hardened SQL database. Some use file-based encrypted storage.
Access Protocol Proprietary protocol or REST API. All access authenticated and encrypted.
Hardening Minimal attack surface‚Äîno unnecessary services, firewall rules restrict access to other PAM components only.
Disaster Recovery Replication to secondary site, encrypted backups, documented recovery procedures.

The Vault Is Your Crown Jewels

The vault contains credentials to your most sensitive systems. Its security is paramount. Best practices include: dedicated hardened server, network isolation, HSM for master key protection, strict change control, and regular security assessments.

Web Portal / Access Interface

The Web Portal (also called PVWA, Web Console, or Access Portal) is the primary interface through which users interact with PAM. It's where users log in, search for systems, request access, and launch sessions.

Key Functions

1

User Authentication

Integrates with identity providers (SAML, OIDC) for SSO and enforces MFA requirements. Validates user identity before allowing any access.

2

Resource Discovery & Search

Provides a searchable catalog of systems and accounts the user is authorized to access. Users can filter by platform, location, or custom tags.

3

Access Request Workflow

Guides users through requesting access, collecting justification, ticket numbers, and routing to approvers when required by policy.

4

Session Launch

Initiates connections to target systems through the session manager. May launch browser-based sessions or native clients.

5

Administration

Provides interfaces for PAM administrators to manage accounts, safes, policies, and review audit logs.

Deployment Considerations

  • Load Balancing: Multiple web portal instances behind a load balancer for high availability
  • SSL/TLS: Always HTTPS with valid certificates; consider certificate pinning for sensitive deployments
  • Network Placement: Typically in a DMZ or internal network segment accessible to users
  • Session Timeout: Configurable idle and absolute session timeouts for security

Privileged Session Manager (PSM)

The Privileged Session Manager is perhaps the most powerful component of PAM. It acts as a proxy between users and target systems, enabling credential injection and session recording without users ever seeing passwords.

Session Types

RDP Sessions (Windows)

Remote Desktop connections to Windows servers. PSM acts as an RDP gateway, connecting to targets on behalf of users.

SSH Sessions (Linux/Unix)

Secure Shell connections to Linux, Unix, and network devices. Captures command-line activity.

Web Application Sessions

Browser-based access to web consoles (vSphere, AWS Console, network device GUIs) with credential injection.

Database Sessions

Connections to databases (SQL Server, Oracle, MySQL) with query logging and data masking options.

Central Password Manager (CPM)

The Central Password Manager (also called Password Rotation Engine or Credential Manager) is responsible for automatically changing passwords on target systems. This is what makes PAM "active" rather than just a password vault.

What It Does

1

Scheduled Rotation

Automatically changes passwords based on schedule (daily, weekly, monthly) to meet compliance requirements and limit credential exposure time.

2

Post-Use Rotation

Changes passwords immediately after a user session ends, ensuring the credential they used is no longer valid.

3

Password Generation

Creates strong, random passwords that meet complexity requirements for each target platform.

4

Verification

After rotation, verifies the new password works by testing authentication to the target system.

5

Reconciliation

If a password gets out of sync (changed manually outside PAM), can use a reconciliation account to reset it.

Platform Support

CPM uses platform-specific plugins (connectors) to change passwords on different types of systems:

  • Windows: Local accounts, domain accounts, service accounts, scheduled task accounts
  • Linux/Unix: Local accounts, SSH keys, sudoers
  • Databases: SQL Server, Oracle, MySQL, PostgreSQL, MongoDB
  • Network Devices: Cisco, Juniper, Palo Alto, F5, Checkpoint
  • Cloud Platforms: AWS IAM, Azure AD, GCP service accounts
  • Applications: SAP, Oracle EBS, custom applications via SDK

Discovery Engine

The Discovery Engine scans your environment to find privileged accounts that should be managed by PAM. You can't protect what you don't know exists.

Discovery Methods

Network Scanning

Scans IP ranges to discover systems and identify local administrator accounts on Windows, root on Linux/Unix.

Directory Integration

Queries Active Directory to find members of privileged groups (Domain Admins, Enterprise Admins, etc.).

Cloud Discovery

Connects to cloud platforms (AWS, Azure, GCP) to discover IAM users, roles, and service accounts with elevated permissions.

Database Scanning

Connects to database servers to discover database accounts with DBA or elevated privileges.

Continuous vs One-Time Discovery

Discovery should be ongoing, not just a one-time activity. New systems are deployed, new accounts are created, and privileged access changes over time.

  • Initial Discovery: Full scan of the environment during PAM deployment
  • Scheduled Scans: Regular scans (daily/weekly) to find new accounts
  • Event-Driven: Integration with provisioning systems to onboard accounts as they're created
  • Alerting: Notifications when new privileged accounts are discovered outside PAM

Privileged Threat Analytics (PTA)

The Privileged Threat Analytics engine uses behavioral analysis and machine learning to detect suspicious privileged activity that might indicate a compromised account or insider threat.

What It Detects

Detection Type Example Scenarios
Unusual Access Times A user who normally works 9-5 accessing systems at 3 AM
Unusual Locations Access from a new country or IP address never seen before
Unusual Targets A user accessing systems they've never accessed before
Unusual Commands Running reconnaissance commands, downloading hacking tools
Credential Theft Indicators Dumping password hashes, accessing credential stores
Privilege Escalation Attempts to gain higher privileges than authorized
Data Exfiltration Large file transfers, access to unusual volumes of data

How It Works

1

Baseline Creation

Learns normal behavior patterns for each user and account over time‚Äîwhat systems they access, when, from where.

2

Real-Time Analysis

Monitors ongoing activity against the baseline, scoring each action for risk.

3

Alert Generation

When risk scores exceed thresholds, generates alerts for security team investigation.

4

SIEM Integration

Sends alerts and risk scores to SIEM for correlation with other security data.

Connector Framework

The Connector Framework provides the integration layer that allows PAM to communicate with different types of target systems. Each target platform requires a specific connector (also called a plugin or platform).

What Connectors Do

  • Authentication: Know how to log in to the target platform
  • Password Change: Know the specific commands/APIs to change passwords
  • Verification: Can test that credentials work
  • Discovery: Can enumerate accounts on the platform
  • Session Support: Enable session manager to connect to the platform

Common Connector Categories

Category Examples
Operating Systems Windows (local/domain), Linux, Unix (AIX, Solaris, HP-UX), macOS
Databases SQL Server, Oracle, MySQL, PostgreSQL, MongoDB, DB2, Sybase
Network Devices Cisco IOS/NX-OS, Juniper, Palo Alto, F5, Checkpoint, Arista
Cloud Platforms AWS IAM, Azure AD, GCP IAM, Oracle Cloud
Virtualization VMware vSphere, Microsoft Hyper-V, Nutanix
SaaS Applications Salesforce, ServiceNow, Workday, custom web applications
Security Devices Firewalls, load balancers, HSMs, PKI systems

Custom Connectors

Most PAM vendors provide SDKs for building custom connectors when out-of-box connectors don't exist for your specific platforms. This is common for legacy systems or proprietary applications.

How Components Work Together

Let's trace through a complete scenario to see how all these components interact.

Scenario: User Accesses Production Database

1

User ‚Üí Web Portal

DBA opens browser, navigates to PAM portal, authenticates via SSO (redirected to identity provider), completes MFA.

2

Web Portal ‚Üí Vault

Portal queries vault for accounts the user is authorized to access. Vault returns list based on access policies.

3

User Selects Database

User searches for and selects the production Oracle database, clicks "Connect." Portal records the request with justification.

4

Web Portal ‚Üí Session Manager

Portal instructs session manager to create a session. Session manager requests credential from vault.

5

Session Manager ‚Üí Target

Session manager uses Oracle connector to establish database connection, injecting the retrieved password.

6

Session Recording

Session manager begins recording all SQL commands and query results. User works through SQL Developer launched via PAM.

7

Analytics Monitoring

Analytics engine monitors the session in real-time, comparing activity to user's baseline behavior. No anomalies detected.

8

Session Ends

User closes SQL Developer. Session manager finalizes recording and stores it.

9

Password Manager ‚Üí Target

Based on policy, CPM connects to Oracle database using the Oracle connector and changes the DBA password to a new random value.

10

Password Manager ‚Üí Vault

CPM stores the new password in the vault, encrypted. The old password is no longer valid anywhere.
