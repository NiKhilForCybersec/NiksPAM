Use Case 1: Windows Server Administration

The Scenario

Your company has 500 Windows servers across multiple data centers. Twenty IT staff need administrative access for patching, troubleshooting, and configuration. Currently, everyone knows the local Administrator password (which is the same on all servers), and people RDP directly to servers using their personal privileged accounts.

The Problems

  • Same password on all 500 servers—one compromised means all compromised
  • No record of who did what on which server
  • When someone leaves, password might not be changed
  • Credentials cached on admin workstations (Mimikatz risk)
  • No visibility into admin activity

PAM Solution

1

Onboard All Local Admin Accounts

PAM discovers and takes ownership of the local Administrator account on each server. Each server gets a unique, randomly generated 25-character password.

2

Configure Session Proxy

Admins connect through PAM's RDP proxy. They never see or type the local admin password. PAM injects credentials automatically.

3

Enable Session Recording

Every RDP session is recorded. You can see exactly what each admin did—every click, every command.

4

Set Rotation Policy

Passwords rotate daily or after each use. Even if someone somehow captures a password, it's invalid within hours.

Results

  • Lateral movement blocked: Compromise of one server doesn't spread
  • Complete audit trail: Every session recorded and searchable
  • Credentials protected: Never on admin workstations
  • Offboarding simplified: Disable PAM access; credentials already rotated

Implementation Tip

Start with a pilot group of non-critical servers. Let admins adjust to the new workflow before rolling out to production. The biggest challenge is often cultural—admins are used to knowing passwords.

Use Case 2: Database Administrator Access

The Scenario

You have 50 production databases (Oracle, SQL Server, PostgreSQL) containing customer data, financial records, and PII. Six DBAs need access for maintenance, performance tuning, and troubleshooting. Compliance requires you to prove who accessed what data and when.

The Problems

  • DBAs know all database passwords (sa, sys, postgres)
  • No way to know which queries a DBA ran
  • Compliance auditors ask "who accessed customer data?" and you can't answer
  • DBAs can access any data, anytime, with no oversight
  • Shared credentials mean no accountability

PAM Solution

1

Vault Database Credentials

All privileged database accounts (sa, sys, oracle, etc.) are onboarded to PAM. Unique, complex passwords are set and rotated.

2

Deploy Database Session Proxy

DBAs connect through PAM's database proxy. PAM supports native database clients (SQL Developer, SSMS) through the proxy.

3

Enable SQL Recording

Every SQL statement is captured. You can see exactly which tables were queried, what data was selected, what was modified.

4

Set Access Policies

Production database access requires manager approval. Sessions limited to 4-hour windows. Automatic disconnect after idle timeout.

Advanced: SQL Command Filtering

Some PAM solutions can filter SQL commands in real-time:

  • Block: DROP TABLE, TRUNCATE, DELETE FROM ... WHERE 1=1
  • Alert: SELECT * FROM customers (bulk data access)
  • Allow: Normal maintenance commands

Results

  • Full query audit: Every SQL statement logged with DBA identity
  • Compliance ready: "Show me all queries against customer table in Q4" is now possible
  • Dangerous commands blocked: Accidental or malicious bulk deletes prevented
  • Data access visibility: Know exactly who accessed what data

Use Case 3: Third-Party Vendor Access

The Scenario

You use a third-party vendor to manage your SAP system. Their consultants need occasional administrative access for upgrades, patches, and troubleshooting. You also have managed service providers who handle your network devices and an external security firm for penetration testing.

The Problems

  • Vendors have VPN access 24/7 even though they rarely need it
  • You gave them credentials that haven't been changed in years
  • No visibility into what vendors do on your systems
  • If the vendor is breached, attackers can use their access to your systems
  • Audit asks "who accessed SAP?" and you have to say "someone at the vendor, we don't know who"

PAM Solution

1

Individual Vendor Accounts

Each vendor consultant gets their own PAM account ([email protected]). No shared "vendor" accounts. You know exactly who is accessing your systems.

2

Just-in-Time Access

Vendors have no standing access. They must request access for each engagement. Request includes: what system, why, how long needed.

3

Approval Workflow

Your internal team (system owner) approves each request. You can see who from the vendor is requesting and why before granting access.

4

Session Recording + Time Limits

All vendor sessions are recorded. Access automatically expires after the approved window (e.g., 4 hours, 1 day). No VPN needed—vendors access through PAM's web portal.

Results

  • Zero standing privilege: Vendors have no access until approved
  • Individual accountability: Know exactly which person did what
  • No VPN exposure: PAM provides secure access without network-level VPN
  • Automatic expiration: Access removed when work is done
  • Supply chain protection: If vendor is breached, attackers find no standing access

Watch For This

Some vendors will push back on individual accounts and session recording. They may claim confidentiality concerns or "proprietary processes." Stand firm—this is your system, your data, your risk. Professional vendors understand and accept these requirements.

Use Case 4: Cloud Console Access

The Scenario

Your organization uses AWS, Azure, and GCP. Cloud admins access the consoles daily to manage infrastructure. Some developers also need console access for specific tasks. The AWS root account credentials are in a shared document.

The Problems

  • Root/Global Admin accounts are the "keys to the kingdom"
  • If cloud console credentials are phished, attackers get everything
  • Hard to track who did what in cloud consoles
  • Developers may have more access than needed
  • No session recording for cloud web consoles

PAM Solution

1

Federated Access Through PAM

Users authenticate to PAM, then PAM provides federated access to cloud consoles. No direct cloud passwords—PAM handles authentication via SAML/OIDC.

2

Root Account Protection

Cloud root/Global Admin credentials are vaulted in PAM. Access requires dual approval. Credentials rotate after use. MFA hardware token is physically secured.

3

Role-Based Cloud Access

PAM assigns users to appropriate cloud roles via federation. Developers get Developer role (limited). Admins get specific admin roles. Nobody gets Global Admin for day-to-day work.

4

JIT Elevation for High-Privilege

Need Global Admin? Request through PAM. Temporary elevation approved, used, then automatically removed. Session is recorded.

Cloud Console Session Recording

For cloud web consoles, PAM can:

  • Record the browser session (video of what the user sees)
  • Capture clicks and actions
  • Integrate with cloud audit logs for complete picture

Results

  • No direct cloud passwords: Users authenticate through PAM; no passwords to phish
  • Root accounts protected: Require multiple approvers, dual control
  • Least privilege: Users get exactly the cloud role they need
  • Session visibility: Know what happened in cloud console sessions

Use Case 5: DevOps & CI/CD Pipelines

The Scenario

Your development team uses Jenkins for CI/CD, deploys to Kubernetes, and stores code in GitHub. Pipelines need credentials to access databases, cloud APIs, and artifact repositories. Currently, secrets are stored in Jenkins credential manager, environment variables, and config files.

The Problems

  • Secrets scattered across multiple places
  • Jenkins credentials aren't rotated
  • Developers can see production credentials
  • If Jenkins is compromised, all secrets are exposed
  • No audit trail of which pipeline used which credential

PAM/Secrets Management Solution

1

Centralize Secrets in Vault

All secrets moved to a secrets manager (HashiCorp Vault, CyberArk Conjur, AWS Secrets Manager). Jenkins/pipelines retrieve secrets at runtime via API.

2

Pipeline Authentication

Each pipeline authenticates to secrets manager using its own identity (service account, IAM role). Pipeline identity determines which secrets it can access.

3

Dynamic Secrets

Instead of static database credentials, secrets manager generates temporary credentials for each pipeline run. Credentials expire after the pipeline completes.

4

Separation of Environments

Development pipelines can only access dev secrets. Production pipelines access prod secrets. Developers can't retrieve production credentials.

Example: Jenkins Pipeline with Vault

Instead of:

# BAD: Hardcoded in Jenkinsfile
DB_PASSWORD = "SuperSecret123"

The pipeline does:

# GOOD: Retrieved at runtime
DB_PASSWORD = vault.read("secret/prod/database").password
# Password exists only in memory during pipeline execution

Results

  • No secrets in code: Nothing sensitive in repos or config files
  • Automatic rotation: Dynamic secrets are always fresh
  • Environment separation: Dev can't accidentally use prod credentials
  • Audit trail: Know which pipeline accessed which secret, when
  • Breach containment: If Jenkins is compromised, attacker finds no stored secrets

Use Case 6: Network Device Management

The Scenario

You have 200 network devices: Cisco routers and switches, Palo Alto firewalls, F5 load balancers. Network engineers need CLI access for configuration changes. Currently, everyone knows the enable password (which is the same on most devices).

The Problems

  • Shared enable passwords across many devices
  • No record of what commands were executed
  • Passwords rarely changed (device reboots needed for some changes)
  • Some devices still use Telnet (credentials in plaintext)
  • TACACS/RADIUS helps but doesn't provide session recording

PAM Solution

1

Vault All Device Credentials

Enable passwords, admin accounts, and SSH keys for all network devices are onboarded to PAM. Unique passwords per device.

2

SSH/Telnet Proxy

Engineers connect through PAM's terminal proxy. PAM handles SSH/Telnet connection to the device. Engineers never see the actual device password.

3

Command Recording

Every command is captured. You can see exactly what configuration changes were made, by whom, when.

4

Command Filtering (Optional)

Block dangerous commands in real-time: write erase, reload, format. Alert on sensitive commands.

Handling Legacy Protocols

For devices that only support Telnet (credentials in plaintext):

  • PAM proxies the Telnet connection
  • Telnet only happens on the secure PAM-to-device segment
  • User connects to PAM via SSH (encrypted)
  • Plaintext credentials never cross untrusted networks

Results

  • Individual accountability: Know exactly who made each change
  • Change tracking: Every command logged and searchable
  • Credential protection: Engineers don't know device passwords
  • Legacy protection: Even Telnet devices are accessible securely

Use Case 7: Service Account Management

The Scenario

Your environment has 300+ service accounts running Windows services, scheduled tasks, and application pools. Nobody knows exactly which service accounts exist, what they're used for, or when their passwords were last changed. Some service accounts are members of Domain Admins "because it was easier."

The Problems

  • No inventory of service accounts
  • Many are over-privileged (Domain Admin for a backup job?)
  • Passwords never changed (fear of breaking things)
  • No ownership—the person who created it left years ago
  • Attackers target service accounts (Kerberoasting)

PAM Solution

1

Discovery

PAM scans Active Directory and local systems to find all service accounts. Build inventory: account name, what it's running, where it's used, group memberships.

2

Assign Ownership

Each service account is assigned an owner (application team, system owner). Owner is responsible for the account and must justify its permissions.

3

Right-Size Permissions

Review and reduce permissions. That backup service account doesn't need Domain Admin—it needs backup operator rights on specific servers.

4

Onboard to PAM

Service account passwords are vaulted. PAM sets complex random passwords. Applications retrieve credentials from PAM via API (or PAM coordinates rotation with the service).

5

Automated Rotation

Passwords rotate on schedule (30-90 days). PAM coordinates: update password in AD → update service configuration → restart service → verify running.

The Rotation Challenge

Service account rotation is tricky because you must update the password everywhere it's used:

  • Active Directory (the account itself)
  • Windows Service configuration
  • Scheduled Task configuration
  • IIS Application Pool
  • Application config files

PAM solutions handle this through:

  • Service mapping: Know everywhere the account is used
  • Coordinated rotation: Update all locations in sequence
  • Verification: Confirm services restart successfully
  • Rollback: If something fails, restore previous password

Results

  • Complete inventory: Know every service account and its purpose
  • Least privilege: Accounts have only necessary permissions
  • Regular rotation: No more "this password hasn't changed since 2015"
  • Kerberoasting defense: Long random passwords can't be cracked

Use Case 8: Emergency Break-Glass Access

The Scenario

It's 3 AM on Sunday. Your primary authentication systems are down. The on-call admin needs to access critical infrastructure to fix the outage, but they can't authenticate normally. You need a secure emergency access path that doesn't compromise security.

The Problems

  • If PAM is down, how do admins access systems?
  • Emergency accounts that bypass normal controls are risky
  • If emergency credentials are too easy to use, they'll be abused
  • If they're too hard to use, emergencies become crises

PAM Solution: Break-Glass Procedures

1

Dedicated Emergency Accounts

Create emergency accounts that are never used for normal operations. These accounts exist specifically for break-glass scenarios.

2

Secure Credential Storage

Emergency credentials stored outside PAM (so they work when PAM is down): physical safe, split-knowledge (two halves held by different people), or sealed envelope with tamper evidence.

3

Detection and Alerting

Any use of emergency accounts triggers immediate alerts to security team, management, and audit. This ensures accountability even without PAM.

4

Post-Use Rotation

After any emergency account use, credentials are immediately rotated. Full incident review: was the use justified? What was done?

Break-Glass Models

Model How It Works Pros/Cons
Physical Safe Passwords in sealed envelopes in a physical safe Simple, works offline. Requires physical presence.
Split Knowledge Password split between two people (e.g., first half + second half) Requires two people. Protects against single person going rogue.
M-of-N 3 of 5 keyholders must combine their parts to reconstruct password High security. More complex coordination.
Secondary PAM Separate PAM instance for emergency access only Modern approach. Requires HA for secondary PAM.

Results

  • Resilience: Critical systems accessible even when PAM is down
  • Accountability: Emergency use is detected and investigated
  • Security: Emergency credentials are not a backdoor for attackers
  • Testing: Regular drills ensure the procedure works

Critical: Test Your Break-Glass

A break-glass procedure you've never tested is a procedure that might not work. Conduct quarterly drills: Can you actually access the safe? Do the credentials still work? Do alerts fire? How long does it take? Discovering problems during an actual emergency is too late.

Summary

These use cases show PAM's versatility: from Windows server management to database access, vendor control to cloud security, DevOps pipelines to network devices. The common thread is controlled, monitored, accountable access to privileged resources. Start with your highest-risk use case, prove value, then expand.
