Types of Privileged Accounts

Understanding the different types of privileged accounts in your environment is the first step to managing them effectively. This page categorizes privileged accounts by type, risk level, and management requirements.

Defining Privileged Access

A privileged account is any account with access rights that exceed those of a standard user. This elevated access can take many forms—administrative rights, access to sensitive data, ability to change configurations, or authority to create other accounts.

The Privilege Spectrum

Privilege isn't binary—it's a spectrum. A database user with SELECT-only access has some privilege. A DBA with full admin rights has much more. PAM focuses on accounts with significant risk potential, but the definition of "significant" varies by organization and regulatory context.

Common Privileged Access Types

System Administration

Install software, change configurations, manage users, access all files on a system.

Data Access

Query, modify, or export sensitive data—customer records, financial data, PII.

Security Controls

Modify security policies, create accounts, change access permissions, disable logging.

Infrastructure

Network configuration, firewall rules, cloud infrastructure provisioning.

Local Administrator Accounts

Local administrator accounts have full control over a single system—they can install software, modify the OS, access all files, and create other local users.

Common Examples

  • Windows: Built-in Administrator, custom local admin accounts
  • Linux/Unix: root account, accounts with sudo privileges
  • macOS: root, admin group members

Risks

Risk Description
Shared credentials Same password used across many systems enables lateral movement
No accountability Multiple people using same account makes attribution impossible
Rarely rotated Passwords often unchanged for years
Pass-the-hash Credential theft from one system enables access to others

PAM Approach

  • Unique passwords: Each system gets a unique local admin password
  • Automatic rotation: Passwords changed regularly (daily, weekly, or after each use)
  • Vaulted storage: Passwords stored in PAM, not known to users
  • Check-out workflow: Users request access, usage is logged

Microsoft LAPS

Microsoft's Local Administrator Password Solution (LAPS) provides basic local admin password management through Active Directory at no cost. It's a good starting point, though enterprise PAM solutions offer more features like session recording and workflow approvals.

Domain/Directory Accounts

Domain administrator accounts have elevated rights across the entire directory service—Active Directory, Azure AD, LDAP directories. These are among the most powerful accounts in any organization.

Windows Active Directory Tiers

Account Type Access Level Risk
Domain Admin Full control of the domain Critical
Enterprise Admin Full control of the forest (all domains) Critical
Schema Admin Can modify AD schema (permanent changes) Critical
Account Operators Create/modify user accounts and groups High
Server Operators Administrative access to domain controllers High
Backup Operators Backup files, including reading all data High

PAM Approach

  • JIT access: No standing Domain Admin membership—add to group only when needed
  • Dedicated accounts: Separate admin accounts from regular user accounts
  • Tier isolation: Tier 0 admin accounts only used on Tier 0 systems
  • Session recording: Full recording of all domain admin sessions
  • Multi-person approval: Require approval for the most sensitive access

Service Accounts

Service accounts are non-human accounts used by applications, services, and automated processes. They often have significant privileges and are frequently overlooked in security programs.

Characteristics

  • Run application services (web servers, databases, middleware)
  • Execute scheduled tasks and batch jobs
  • Connect between systems (API integrations)
  • Often have more privileges than necessary
  • Passwords rarely changed (fear of breaking applications)

Types of Service Accounts

Type Platform Notes
Windows Service Account Windows Runs Windows services (Local System, custom accounts)
Managed Service Account (MSA) Windows/AD AD-managed password, single server
Group Managed Service Account (gMSA) Windows/AD AD-managed password, multiple servers
Linux daemon accounts Linux Run services (apache, mysql, postgres, etc.)
Scheduled task accounts All platforms Run cron jobs, Windows Task Scheduler

PAM Approach

  • Inventory: Discover all service accounts and document their purpose
  • Ownership: Assign an owner responsible for each service account
  • Least privilege: Review and reduce permissions to minimum necessary
  • Password rotation: Coordinate rotation with application teams
  • API access: Use PAM APIs for applications to retrieve credentials

Application Accounts

Application accounts are administrative accounts within applications—not OS-level accounts, but accounts that manage the application itself.

Examples

  • Web application admin: WordPress admin, Drupal admin, CMS administrators
  • Business application admin: SAP basis, Salesforce admin, ServiceNow admin
  • Virtualization admin: vCenter administrator, Hyper-V admin
  • Security tool admin: SIEM admin, firewall admin, EDR admin
  • DevOps tools: Jenkins admin, GitLab admin, Kubernetes admin

Challenges

Application Account Challenges
  • Non-standard authentication: Many apps don't integrate with directory services
  • Web-based access: Traditional PAM session proxy may not work
  • Vendor defaults: Default admin passwords often left unchanged
  • Lack of visibility: Often not tracked in identity systems

PAM Approach

  • Web session recording: Use PAM's web application connector to record sessions
  • Password vaulting: Store application admin passwords in vault
  • SSO integration: Where possible, federate app access through IdP
  • Rotation via API: Use application APIs to rotate passwords where available

Database Accounts

Database administrator (DBA) accounts have direct access to organizational data and database infrastructure. They represent significant risk due to their data access.

Common Database Accounts

Database Admin Account Capabilities
SQL Server sa (system administrator) Full database server control
Oracle SYS, SYSTEM Full database control, data dictionary
MySQL/MariaDB root Full database server control
PostgreSQL postgres (superuser) Full database cluster control
MongoDB root, userAdminAnyDatabase Full database control

PAM Approach

  • Session recording: Record all DBA sessions, capture SQL queries
  • Query logging: Log all commands for audit and analysis
  • Credential injection: DBAs connect through PAM, never see passwords
  • Command blocking: Optionally block dangerous commands (DROP, DELETE without WHERE)
  • Time-limited access: JIT access for maintenance windows

Network Device Accounts

Network device accounts manage routers, switches, firewalls, load balancers, and other network infrastructure. Compromise of these accounts can enable network-wide attacks.

Examples

  • Cisco IOS enable/privilege 15 accounts
  • Palo Alto firewall admin
  • F5 BIG-IP admin
  • Juniper root
  • Wireless controller admin

Challenges

  • Legacy protocols: Many devices use Telnet, which is unencrypted
  • Limited authentication: May not support RADIUS/TACACS+ integration
  • Shared accounts: Often one admin account shared by entire team
  • Firmware limitations: Password complexity/rotation may be limited

PAM Approach

  • Proxy all access: All network device access goes through PAM (even Telnet becomes encrypted)
  • Session recording: Record all configuration changes
  • Individual accountability: Users authenticate to PAM, then PAM uses shared credentials
  • Automatic rotation: PAM rotates device passwords on schedule

Cloud IAM Accounts

Cloud IAM accounts with administrative privileges can provision infrastructure, access data, and modify security controls across cloud environments.

Cloud Admin Account Types

Cloud Admin Roles Risk Level
AWS Root account, IAM users with AdministratorAccess Critical
Azure Global Administrator, Subscription Owner Critical
GCP Organization Admin, Project Owner Critical

PAM Approach

  • Federated access: No standing cloud admin accounts—federate from IdP
  • JIT role elevation: Use cloud-native PIM (Azure AD PIM, AWS temporary credentials)
  • Root account protection: Lock down cloud root/billing accounts with MFA and no regular use
  • Session recording: Record cloud console access via web session recording
  • API key management: Vault and rotate API keys/access keys

SSH Keys

SSH keys provide passwordless authentication to Linux/Unix systems. While more secure than passwords in some ways, unmanaged SSH keys create significant risk.

SSH Key Risks

  • No expiration: SSH keys don't expire by default
  • Proliferation: Keys copied between systems, often without tracking
  • No central visibility: Hard to know what keys exist and who has them
  • Difficult revocation: Must remove from all authorized_keys files
  • Shared keys: Same key pair used by multiple people

PAM Approach

  • Vault private keys: Store SSH private keys in PAM vault
  • Certificate-based access: Issue short-lived SSH certificates instead of static keys
  • Session proxy: Users connect through PAM, which handles SSH authentication
  • Key discovery: Scan systems to inventory existing authorized_keys
  • Key rotation: Regularly rotate key pairs

Emergency/Break-Glass Accounts

Break-glass accounts are emergency access accounts used when normal access paths fail—during outages, when PAM is unavailable, or for disaster recovery.

Characteristics

  • Highest privilege level (Domain Admin, root, cloud admin)
  • Rarely used—only in true emergencies
  • Password may be stored in physical safe or split among custodians
  • Usage should trigger immediate alerts

Best Practices

Break-Glass Account Management
  • Secure storage: Password in sealed envelope in physical safe, or M-of-N split
  • Detection: Alert immediately on any use of break-glass accounts
  • Rotation after use: Rotate password after every use
  • Periodic testing: Test break-glass access works (quarterly/annually)
  • Documentation: Clear procedures for when and how to use
  • Post-incident review: Review every use—was it justified?

Account Discovery

You can't manage what you don't know exists. Privileged account discovery is the process of finding all privileged accounts across your environment.

Discovery Methods

  • AD enumeration: Query Active Directory for privileged group members
  • Network scanning: Connect to systems and enumerate local accounts
  • Agent-based: Deploy agents that inventory accounts on each system
  • Cloud API: Query cloud IAM APIs for privileged identities
  • SSH key scanning: Find authorized_keys files across Linux systems
  • Application integration: Query applications for admin accounts

Discovery Outputs

A good discovery process produces:

  • Complete inventory of privileged accounts
  • Account classification by type and risk
  • Identification of orphaned accounts (no owner)
  • Detection of shared credentials
  • Mapping of accounts to systems/applications

Key Takeaways

Privileged accounts take many forms—local admins, domain admins, service accounts, application accounts, DBA accounts, network device accounts, cloud IAM, and SSH keys. Each type has specific risks and requires tailored PAM controls. Start with discovery to understand your environment, then prioritize based on risk.
