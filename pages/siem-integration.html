SIEM Integration

Connecting PAM and KMS to your Security Information and Event Management (SIEM) platform enables real-time monitoring, threat detection, and compliance reporting for privileged access.

Why Integrate with SIEM?

PAM and KMS generate valuable security telemetry. Integrating with your SIEM provides:

Real-Time Visibility

See privileged access events as they happen, not hours or days later.

Threat Detection

Correlate PAM events with other security data to detect attacks.

Compliance Reporting

Generate audit reports showing who accessed what, when, and why.

Correlation

Combine PAM data with endpoint, network, and cloud logs for full picture.

The Value of Correlation

PAM events alone are useful. But combined with endpoint data? Powerful. Example: PAM shows user checked out Domain Admin credential at 3 AM. EDR shows that same user's workstation ran Mimikatz 5 minutes later. Individually interesting; together, clearly an attack.

What Events to Send

Don't send everything—focus on security-relevant events. Here's a prioritized list:

High Priority (Always Send)

Event Type Why It Matters Detection Use Cases
Break-glass account usage Emergency accounts used means something unusual Immediate investigation trigger
Authentication failures Could indicate brute force or credential stuffing Multiple failures = alert
Session terminations (admin-initiated) Security team killed a session Incident response correlation
Credential checkout for high-tier accounts Domain Admin, root, cloud admin access Track Tier 0/1 access
Policy changes Someone modified security settings Detect unauthorized changes
User provisioning/deprovisioning New access granted or removed Detect privilege creep, insider prep

Medium Priority (Recommended)

Event Type Why It Matters
All credential checkouts Track who accessed what credentials
Session start/end Know when privileged sessions occur
Password rotation results Detect rotation failures that need attention
Approval workflow events Who approved access, when
KMS key usage Track encryption operations

Lower Priority (Nice to Have)

  • Successful authentications (high volume, less actionable)
  • Discovery scan results
  • Health monitoring events
  • Scheduled maintenance events

Splunk Integration

Splunk is one of the most common SIEMs in enterprise environments. Here's how to integrate:

Integration Methods

Method How It Works When to Use
Syslog PAM sends logs to Splunk via syslog (UDP/TCP) Simple setup, any PAM vendor
HTTP Event Collector (HEC) PAM POSTs JSON events directly to Splunk Better performance, structured data
Splunk Add-on/App Vendor-specific app with parsing and dashboards Best experience, CyberArk/BeyondTrust have apps
Universal Forwarder Agent on PAM server reads log files When direct integration not available

Sample Splunk Queries

Find All Break-Glass Usage

index=pam sourcetype=cyberark:pam
| where like(safe, "%BreakGlass%") OR like(safe, "%Emergency%")
| table _time, user, safe, account, target, action
| sort -_time

Authentication Failures Over Time

index=pam sourcetype=cyberark:pam action="Logon" result="Failure"
| timechart count by user
| where count > 5

High-Value Account Access

index=pam sourcetype=cyberark:pam action="Retrieve"
| where match(account, "(?i)admin|root|domain|enterprise")
| stats count by user, account, target
| sort -count

Unusual Access Patterns (Outside Business Hours)

index=pam sourcetype=cyberark:pam action="Connect"
| eval hour=strftime(_time, "%H")
| where hour < 6 OR hour > 20
| table _time, user, account, target
| sort -_time

Splunk Alert Examples

Alert: Break-Glass Account Used

Trigger: Any event from break-glass/emergency safe
Severity: High
Action: Page on-call security analyst, create incident ticket

Alert: Multiple Auth Failures

Trigger: More than 5 failed logins from same user in 10 minutes
Severity: Medium
Action: Email security team, consider account lockout

Microsoft Sentinel Integration

For organizations using Microsoft's cloud-native SIEM:

Integration Options

  • Data Connector: Some PAM vendors have native Sentinel connectors
  • Common Event Format (CEF): Send via Log Analytics agent
  • Azure Monitor Agent: Collect from syslog or log files
  • Logic Apps: Pull from PAM API, push to Sentinel
  • Azure Event Hub: Stream events through Event Hub

Sample KQL Queries

Privileged Session Activity

PAMEvents
| where ActionType == "SessionConnect"
| summarize SessionCount=count() by User, TargetHost, bin(TimeGenerated, 1h)
| order by SessionCount desc

Credential Access by Risk Level

PAMEvents
| where ActionType == "CredentialRetrieve"
| extend AccountRisk = case(
    AccountName contains "admin" or AccountName contains "root", "High",
    AccountName contains "service", "Medium",
    "Low"
)
| summarize AccessCount=count() by AccountRisk, User
| render piechart

Failed Operations

PAMEvents
| where Result == "Failure"
| summarize FailureCount=count() by ActionType, User, TargetHost
| where FailureCount > 3
| order by FailureCount desc

Sentinel Analytics Rules

Rule: Unusual Access to Domain Controller

PAMEvents
| where TargetHost contains "DC" or TargetHost contains "DomainController"
| where TimeGenerated > ago(24h)
| join kind=leftanti (
    PAMEvents
    | where TimeGenerated between (ago(30d) .. ago(24h))
    | distinct User, TargetHost
) on User, TargetHost
| project TimeGenerated, User, TargetHost, ActionType

This finds users accessing domain controllers who haven't done so in the past 30 days—potential suspicious activity.

Detection Rules & Use Cases

Here are specific detection scenarios to implement:

1. Credential Theft Detection

Pattern Indicators Response
Mimikatz Activity PAM credential checkout + EDR detects Mimikatz on same user's workstation Terminate session, rotate credential, investigate
Pass-the-Hash Attempt PAM credential checkout + Windows Event 4624 Type 9 from unusual source Block source IP, rotate credential
Golden Ticket Domain Admin access via PAM + Kerberos TGT from unexpected time/location Revoke tickets, rotate krbtgt twice

2. Insider Threat Detection

Pattern Indicators Response
Data Exfiltration Prep User accesses more systems than usual + DLP detects large downloads Alert manager, enable enhanced monitoring
Privilege Escalation User requests access to systems outside their normal scope Require additional approval, flag for review
Off-Hours Activity Privileged access at unusual times (3 AM, weekends) Correlate with ticket system, investigate if no ticket

3. Ransomware Precursor Detection

Pattern Indicators Response
Backup System Access Unusual user accesses backup admin credentials High alert—ransomware targets backups first
Mass Credential Checkout Single user checks out many credentials in short time Suspicious reconnaissance—terminate access
GPO Access Non-standard user accesses Group Policy admin credentials Ransomware uses GPO for mass deployment—investigate immediately

4. KMS Security Events

Event What It Means Response
Key deletion Someone deleted a cryptographic key Verify authorized, check if data recoverable
Unusual decrypt volume Massive decryption requests (data exfil?) Investigate source application
Key policy change Permissions on key were modified Verify change was authorized
Cross-region key access Key used from unexpected region Could be attack from compromised region

Dashboards and Reporting

Executive Dashboard Elements

  • Total privileged sessions (24h, 7d, 30d trends)
  • Unique privileged users active in period
  • High-risk access events (Tier 0/1 systems)
  • Break-glass usage (should be near zero)
  • Policy violations (denied access attempts)
  • Password rotation compliance (% on schedule)

Security Operations Dashboard

  • Real-time session map (who's connected where)
  • Authentication failure trend
  • Alerts by severity (open/acknowledged/resolved)
  • Top users by access volume
  • Top systems by access volume
  • Geographic access map (for remote/vendor access)

Compliance Reports

  • Access certification: All access granted in period
  • Privileged account inventory: All managed accounts
  • Rotation compliance: Which accounts rotated, which overdue
  • Session recordings: List of recorded sessions for audit
  • Access by system: Who accessed each critical system

SOAR Automation

Security Orchestration, Automation, and Response (SOAR) platforms can automate responses to PAM events:

Automated Response Playbooks

Playbook: Suspicious Privileged Access

1

Trigger

PAM alert: User accessed Domain Admin credential outside business hours

2

Enrich

Query ITSM: Is there an approved change ticket? Query HR: Is user on call rotation?

3

Decision

If valid ticket exists → log and close. If no ticket → escalate.

4

Respond (if suspicious)

Terminate PAM session via API. Send Teams/Slack message to user asking to confirm. Alert security analyst.

5

Document

Create incident ticket. Attach PAM session recording. Log all actions taken.

Playbook: Credential Rotation Failure

1

Trigger

PAM alert: Password rotation failed for service account

2

Enrich

Get account details. Identify owner from CMDB. Check recent changes to target.

3

Attempt Recovery

If failure was network issue: Wait 5 min, retry via PAM API.

4

Escalate if Failed

Create ticket assigned to account owner. Include troubleshooting steps. Set SLA for resolution.

Key Takeaways

SIEM integration transforms PAM from a control tool into a detection tool. Prioritize high-value events (break-glass, failures, high-tier access). Build detection rules that correlate PAM data with other sources. Create dashboards for both executives and SOC analysts. Automate response with SOAR for faster reaction to threats.
