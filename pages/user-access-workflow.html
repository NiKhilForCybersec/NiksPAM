Why This Matters

Understanding this workflow helps you: (1) Use PAM correctly as an end user, (2) Troubleshoot access issues, (3) Design PAM policies, and (4) Explain PAM to colleagues and auditors.

Step 1: User Opens PAM Portal

1

What Happens

The user opens their web browser and navigates to the PAM portal URL (e.g., https://pam.yourcompany.com).

What the User Sees

The PAM portal is a web-based interface. It looks like a login page initially, then after authentication shows a dashboard with available systems.

Common PAM Portal Features
  • Search Bar: Find systems by name, IP address, or tag
  • Favorites: Quick access to frequently used systems
  • Recent Sessions: History of your past access
  • Request Queue: Pending access requests and their status
  • Categories: Systems organized by type (servers, databases, network devices)

Behind the Scenes

When you access the portal, the PAM system:

  • Serves the web interface from a secure web server
  • Uses HTTPS encryption for all communications
  • May check if your workstation meets security requirements
  • Prepares to redirect you to authentication

Step 2: Authentication (Who Are You?)

2

What Happens

The PAM portal redirects you to your organization's identity provider (IdP) to prove who you are. This typically involves Single Sign-On (SSO) and Multi-Factor Authentication (MFA).

Types of Multi-Factor Authentication

MFA Type How It Works Security Level

TOTP Code Enter a 6-digit code from an authenticator app High
Hardware Token Insert/tap a physical key (YubiKey) Very High

Why MFA is Critical for PAM

Privileged access is so sensitive that MFA is mandatory. Even if an attacker steals your password, they can't access PAM without the second factor. Many organizations require stronger MFA (hardware tokens) for accessing the most critical systems.

Step 3: Finding the Target System

3

What Happens

After authentication, the PAM portal shows you a list of systems you're authorized to access. You search for or browse to find the system you need.

What You Can See

The PAM portal only shows you systems your role is allowed to access. A database administrator sees database servers; a network engineer sees routers and switches. This is called role-based access control (RBAC).

Information Displayed for Each System

Typical System Information
  • System Name: Hostname or friendly name (e.g., "PROD-DB-01")
  • IP Address: Network address of the system
  • Platform: Windows, Linux, database type, etc.
  • Account Options: Which privileged accounts you can use (admin, root, sa)
  • Connection Type: RDP, SSH, database client, web
  • Environment: Production, staging, development
  • Approval Required: Whether you need approval to access

Search and Filter Options

Most PAM portals let you:

  • Search by name, IP, or keyword
  • Filter by platform (Windows, Linux, Network)
  • Filter by environment (Production, Non-Production)
  • Filter by location or data center
  • Save favorites for quick access

Step 4: Requesting Access

4

What Happens

You click on the target system and select the type of access you need. You may need to provide a reason or ticket number for the access.

Types of Access Requests

Interactive Session

Connect directly to the system (RDP, SSH) to perform work interactively. Most common for administrators.

Show Password

Reveal the password so you can use it in a native client. Used when PAM session proxy isn't supported.

Copy Password

Copy credential to clipboard for pasting into an application. Clipboard is typically cleared after a short time.

Information Required for Access Request

Field Description Required?
Target System The server/device you want to access Always
Account Which privileged account to use (e.g., Administrator, root) Always
Connection Type How you want to connect (RDP, SSH, etc.) Usually
Ticket Number Reference to change/incident ticket (e.g., INC-12345) Often
Business Reason Why you need access (free text) Often
Duration How long you need access (e.g., 4 hours) Sometimes

Why Request Justification Matters

Requiring a ticket number and reason creates accountability. It links privileged access to authorized work, making it easy to audit later. If an incident occurs, investigators can see exactly why access was granted.

Step 5: Approval Process

5

What Happens

Depending on the system's sensitivity and your role, your request may be auto-approved or may require one or more approvers to review and approve it.

Approval Scenarios

Scenario Approval Type Example
Low-Risk Access Auto-Approved Developer accessing dev server during business hours
Normal Access Manager Approval Admin accessing production server with ticket
High-Risk Access Multiple Approvers Accessing financial database requires manager + security
Emergency Access Break-Glass (post-audit) Critical outage requiring immediate access

Approval Workflow Steps

A

Request Submitted

Your request enters the approval queue with all the information you provided.

B

Approver Notified

Approver(s) receive email/notification with request details. They can approve from email or the PAM portal.

C

Review and Decision

Approver reviews the reason, ticket, and context. They approve, reject, or request more information.

D

You're Notified

You receive notification that access was approved or denied. If approved, you can now connect.

Time-Bound Approvals

Approvals typically have an expiration. If you request 4 hours of access, you have 4 hours from approval to use it. After that, you'd need to request again. This prevents "stale" approvals from being used days later.

6

What Happens

Connection Methods

Browser-Based (HTML5)

Session runs in your web browser. Most secure option‚Äînothing installed locally, all traffic through PAM.

Native Client

PAM launches your local RDP/SSH client with credentials. Better performance but requires agent installation.

Transparent Proxy

Connect normally but traffic routes through PAM. Users may not even realize PAM is involved.

You Never See the Password

In a properly configured PAM session, you never see or handle the actual privileged credential. PAM retrieves it from the vault and injects it directly into the connection. This means even if your workstation is compromised, the attacker doesn't capture the privileged password.

Step 7: Working in the Session

7

What Happens

You're now connected to the target system with full administrative access. You perform your work while PAM records everything you do.

What Gets Recorded

Recording Type What It Captures Use Case
Video Recording Screen capture like a video, can be played back See exactly what the user saw and did
Keystroke Logging Every key pressed, searchable text Find specific commands or actions
Command Capture SSH commands and output Audit exactly what commands ran
File Transfer Log Files uploaded or downloaded Detect data exfiltration
Window Titles Names of applications/windows opened Quick overview of activity

Session Controls and Restrictions

PAM can enforce policies during your session:

  • Command Restrictions: Block dangerous commands (e.g., rm -rf /)
  • Application Whitelisting: Only allow specific applications
  • File Transfer Limits: Block or limit uploads/downloads
  • Clipboard Control: Prevent copy/paste from session
  • Time Limits: Auto-disconnect after maximum session time
  • Idle Timeout: Disconnect after period of inactivity

Live Monitoring

For high-risk sessions, security team members can:

  • Watch the session in real-time
  • Receive alerts on suspicious commands
  • Send messages to the user
  • Forcibly terminate the session if needed

Assume You're Being Watched

When using PAM, always assume your session is being recorded and could be reviewed. This isn't about distrust‚Äîit's about accountability. Only do things you'd be comfortable explaining to an auditor or your manager.

Step 8: Session Termination

8

What Happens

When you're done, you close the session (log out or close the window). PAM detects the session has ended and completes the recording.

Ways a Session Can End

Termination Type How It Happens Recording Status
Normal Logout User logs out properly Complete recording saved
Window Close User closes browser/client window Complete recording saved
Idle Timeout No activity for X minutes Complete recording saved
Time Limit Reached Maximum session duration exceeded Complete recording saved
Admin Termination Security team ends session manually Recording notes forced termination
Network Failure Connection lost unexpectedly Recording up to disconnect point

Why This Matters

Automatic rotation after use means:

  • Intercepted Credentials Are Useless: If an attacker captured the password during your session, it's already changed
  • No Credential Sharing: Even if you wanted to share the password, you don't know it
  • Reduced Attack Window: Credentials are valid only for the session duration
  • Compliance Satisfied: Meets requirements for regular password changes

Recording Storage and Retention

Session recordings are stored for audit and compliance purposes:

  • Typical Retention: 1-7 years depending on compliance requirements
  • Storage: Encrypted and tamper-proof
  • Indexing: Searchable by user, date, system, and commands
  • Access Control: Only authorized reviewers can watch recordings

Different Access Methods

Not all access goes through interactive sessions. Here are the different ways credentials can be accessed through PAM:

1. Interactive Session (What We Covered)

User connects through PAM portal, session is brokered and recorded.

2. Show Password / Copy Password

Sometimes you need the actual password (e.g., for an application that can't go through PAM proxy). In this case:

  • You request access and get approval
  • PAM displays the password on screen or copies to clipboard
  • Password is automatically rotated after a short time
  • This is logged but there's no session recording

3. API/Application Access

Applications can retrieve credentials programmatically:

  • Application authenticates to PAM via API
  • PAM validates the application identity
  • Credential is returned securely
  • Application uses credential to connect to target

4. SSH Key Checkout

For systems using SSH key authentication:

  • PAM stores SSH private keys in the vault
  • User requests SSH session
  • PAM uses the private key to establish connection
  • User never sees the private key material

You've Mastered the PAM Access Workflow!

You now understand the complete journey from opening a browser to session completion. This knowledge helps you use PAM effectively, troubleshoot issues, and explain the process to others.
